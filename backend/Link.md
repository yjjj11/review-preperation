## 计算机网络高频问答整理

### 1. 客户端怎么验证证书合法性？

**Q:** HTTPS 握手时，客户端如何验证服务端证书是否合法？

**A:**

证书生成过程：
- 首先 CA 会把持有者的公钥、用途、颁发者、有效时间等信息打成一个包，然后对这些信息进行 Hash 计算，得到一个 Hash 值；
- 然后 CA 会使用自己的私钥将该 Hash 值加密，生成 Certificate Signature，也就是 CA 对证书做了签名；
- 最后将 Certificate Signature 添加在文件证书上，形成数字证书；
- 在进行第一次握手的时候，服务端直接把证书传递给客户端。

客户端验证过程：
- 首先客户端会使用同样的 Hash 算法获取该证书的 Hash 值 H1；
- 通常浏览器和操作系统中集成了 CA 的公钥信息，浏览器收到证书后可以使用 CA 的公钥解密 Certificate Signature 内容，得到一个 Hash 值 H2；
- 最后比较 H1 和 H2，如果值相同，则为可信赖的证书，否则则认为证书不可信。
- 假如在 HTTPS 的通信过程中，中间人篡改了证书原文，由于他没有 CA 机构的私钥，所以 CA 公钥解密的内容就不一致。

### 2. 分布式环境下 Session 怎么处理？

**Q:** 分布式环境下，客户端请求经过负载均衡，可能会分配到不同的服务器上，假如一个用户的请求两次没有落到同一台服务器上，那么在新的服务器上就没有记录用户状态的 Session。这时候怎么办呢？

**A:**

可以使用 Redis 等分布式缓存来存储 Session，在多台服务器之间共享 Session 数据，所有服务器都从 Redis 读取 Session，就能解决分布式 Session 问题。

### 3. 如果第三次握手丢了，会怎么样？

**Q:** TCP 三次握手第三次握手 ACK 丢了，双方会怎么处理？

**A:**

- **客户端**：认为连接已经成功，直接发送数据；实际上服务端没有收到 ACK，还没认为连接建立成功。
- **服务端**：会重传 SYN+ACK 多次失败后，放弃连接，accept() 报错；
- 最终服务端收到客户端数据后，发 RST 复位包关闭连接，客户端才知道连接失败，直接进入 CLOSED 阶段。

---

### 4. 第二次握手传回了 ACK，为什么还要传回 SYN？

**Q:** 第二次握手为什么需要同时带 SYN？

**A:**

- **ACK 作用**：确认客户端的 SYN，验证报文来源，完成「确认」功能；
- **SYN 作用**：同步服务端自己的初始序号，任何一方发送数据前都必须同步自己的初始序号。SYN + seq 就是为了把自己的初始序号发给对方同步。

TCP 连接是双向的，双方都需要同步各自的初始序号，所以第二次握手必须同时带 ACK 和 SYN。

### 5. 怎么应对 SYN flood（DDoS）攻击？

**Q:** SYN flood 攻击是什么？怎么防范？

**A:**

**SYN flood**：攻击者大量发送 SYN 包，不回复第三次 ACK，占满服务端半连接队列，让正常用户无法连接。

常见防范方法：

- **syn cookie**：在收到 SYN 包后，服务器根据一定的方法，以数据包的源地址、端口等信息为参数计算出一个 cookie 值作为自己的 SYNACK 包的序列号，回复 SYN+ACK 后，服务器并不立即分配资源进行处理，等收到发送方的 ACK 包后，重新根据数据包的源地址、端口计算该包中的确认序列号是否正确，如果正确则建立连接，否则丢弃该包。实际上可以理解为不提前占半连接队列资源。
- **SYN Proxy 防火墙**：服务器防火墙会对收到的每一个 SYN 报文进行代理和回应，并保持半连接。等发送方将 ACK 包返回后，再重新构造 SYN 包发到服务器，建立真正的 TCP 连接。

### 6. 保活计时器有什么用？

**Q:** TCP 为什么需要保活计时器？

**A:**

解决客户端突然宕机、服务端无限等待的问题：

- 服务端每收到客户端数据，重置保活计时器（默认 2 小时）；
- 超时后，每隔 75 秒发探测报文，连续 10 次无响应则关闭连接；
- 释放资源，避免浪费。

### 7. 为什么挥手的时候客户端收到第二次挥手不能直接关闭？

**Q:** TIME_WAIT 为什么要等 2MSL？直接关闭不行吗？

**A:**

防止旧连接的残留数据包干扰新连接：如果客户端收到服务端的 FIN 报文之后立即关闭连接，但是此时服务端对应的端口并没有关闭，如果客户端在相同端口建立新的连接，可能会导致新连接收到旧连接残留的数据包，导致不可预料的异常发生。

所以 TIME_WAIT 等 2MSL（最长报文寿命），让网络上所有残留的报文都消失，再关闭连接就安全了。

### 8. 说说 TCP 报文首部的格式？

**Q:** TCP 首部包含哪些字段？各有什么用？

**A:**

- **源 / 目的端口号**（16 位 × 2）：定位通信进程，找到对应进程。
- **序号 / 确认号**（32 位 × 2）：序号标识当前报文第一个数据字节在流中的位置；确认号表示期望收到下一字节的序号，确认前序所有数据都收到了。
- **首部长度**（4 位）：单位是 4 字节，最大 60 字节（因为选项最多 40 字节）。
- **标志位**（6 位）：URG/ACK/PSH/RST/SYN/FIN，控制连接状态与数据传输。
- **窗口大小**（16 位）：告诉对方我这边接收缓冲区还有多大，用于流量控制。
- **校验和**（16 位）：校验首部 + 数据，保证传输可靠性。
- **紧急指针**（16 位）：紧急数据偏移，仅 URG=1 时有效，标识紧急数据在哪里。

### 9. 什么是快速恢复？

**Q:** TCP 拥塞控制中的快速恢复是什么作用？

**A:**

快速恢复与快速重传配套使用，收到 3 个重复 ACK 时触发（说明网络丢了一个包，但网络其实还能通，不需要等待 RTO 超时重传）。

进入前先更新参数：`cwnd = cwnd / 2`，`ssthresh = cwnd`。

核心流程：
- `cwnd = ssthresh + 3`（3 个重复 ACK 对应 3 个空闲额度，允许重传丢包 + 利用剩余带宽）
- 每收到 1 个重复 ACK，`cwnd += 1`（后续包成功到达，动态抬升窗口，避免速率过低）
- 收到新数据 ACK 后，`cwnd = ssthresh`，恢复正常拥塞避免（线性增长）

核心作用：温和降速 + 精准利用带宽，避免超时重传的断崖式性能损失，平衡可靠性与吞吐量。

## MySQL 高频问答整理

### 1. MyISAM 与 InnoDB 的区别是什么？

**Q:** MySQL 两个常用存储引擎核心区别？怎么选？

**A:**

最核心区别在于对事务的支持：

- **InnoDB**：支持事务，实现了 ACID 特性，非常适合需要高可靠性的场景，比如在线交易。为了支持事务，InnoDB 提供了**行级锁**和 **MVCC**，在高并发读写下性能表现很好。
- **MyISAM**：不支持事务，使用的是**表级锁**，写操作会锁住整张表，所以在写入并发量大的时候性能会比较差，但它的结构相对简单，读取速度很快，比较适合一些**读多写少**的应用场景。

---

### 2. in 和 exists 区别？

**Q:** SQL 查询中 in 和 exists 怎么选？性能区别是什么？

**A:**

in 和 exists 主要用在子查询中，它们的执行逻辑不同，性能差异取决于数据量分布：

**通用法则：小表驱动大表。**

- 如果**子查询结果集小**，**主查询表大**，那么用 `in` 效率会比较高。因为 in 会先执行子查询，然后把结果缓存起来，再去主查询的表里匹配。
- 如果**子查询表大**，**主查询表小**，那么用 `exists` 会更好。因为 exists 是拿主查询表里的每一行记录，去子查询的表里做一个判断，看是否存在匹配。

另外，对于 `not in` 和 `not exists`，一般推荐使用 `not exists`，因为 `not in` 在很多情况下无法有效利用索引，会导致全表扫描。

---

### 3. 删除表的时候 drop、delete 与 truncate 的区别是什么？

**Q:** 三个删除命令区别在哪里？

**A:**

三个命令删除级别完全不同：

- **delete**：是 DML 语言，它是一行一行地删除记录，可以带 WHERE 子句只删除部分数据。因为是事务性操作，所以 delete 的删除是**可以回滚**的，并且会触发触发器。
- **truncate**：是 DDL 语言，它会直接删除表里的所有数据，保留表结构，不能带 WHERE 子句。它相当于重建了表，所以速度非常快，**不能回滚**，也不会触发触发器。
- **drop**：也是 DDL 语言，它是最彻底的，会直接把整张表，包括表结构、数据、索引、约束等全部从数据库里删除掉，当然也不能回滚。

### 4. MySQL 索引失效常见场景

**Q:** 哪些情况索引会失效？

**A:**

- **范围查询**（> < >= <= between）：并非必然失效，数据量小走索引、数据量超过 20%~30% 时因回表代价过高，优化器会选择全表扫描导致索引失效。
- **字段类型不一致**：隐式类型转换导致索引失效
- **索引列上用函数**：比如 `DATE(create_time)`，会导致索引失效
- **索引列做运算**：`+ - * /` 等运算操作，索引失效
- **OR 连接不同字段**：索引失效（同字段 OR 有效）
- **左模糊查询**：`LIKE '%xx'`，索引失效（`LIKE 'xx%'` 有效）

### 5. 为什么官方建议使用自增长主键作为索引？

**Q:** InnoDB 为什么推荐自增主键？UUID 为什么不好？

**A:**

自增主键插入为顺序追加，仅在页满时新增页，页分裂影响极小；随机主键（如 UUID）插入需要插入页中间，频繁触发页分裂与数据挪动，性能损耗大且容易导致索引碎片化。

### 6. 如何查看 MySQL 死锁日志？怎么解决线上死锁问题？

**Q:** 线上出现死锁了怎么排查解决？

**A:**

**查看死锁日志：**
执行命令：`SHOW ENGINE INNODB STATUS;`
定位日志中 `LATEST DETECTED DEADLOCK` 段，重点关注：
- 涉及的两个事务（TRANSACTION 1/2）
- 事务持有的锁（HOLDS THE LOCK (S)）
- 事务等待的锁（WAITING FOR THIS LOCK）
- MySQL 回滚的哪个事务

**解决死锁问题：**
- **短期应急**：`KILL` 持有锁的 MySQL 线程 ID，快速恢复业务。
- **长期根治**：
  1. 统一加锁顺序：多表操作时，所有事务按相同顺序加锁
  2. 拆分大事务：缩短锁持有时间，降低冲突概率
  3. 优化索引：避免无索引导致的全表扫描和不必要的锁

### 7. 什么是 MySQL 两阶段提交？为什么需要两阶段提交？

**Q:** redo log 和 binlog 为什么需要两阶段提交？流程是什么？

**A:**

两阶段提交是 MySQL 内部为了保证数据一致性而设计的一种分布式事务协调机制，它主要用来协调 InnoDB 的 redo log 和 Server 层的 binlog 这两个独立日志的写入。它把事务的提交分成了两个阶段：

**准备阶段（Prepare）**：当 InnoDB 写完 redo log 后，它不直接提交事务，而是将 redo log 的状态设置为 "prepare"，此时 redo log 是存在 redo log 环形缓冲区中的。

**提交阶段（Commit）**：之后，由执行器去写 binlog。当 binlog 成功写入磁盘后，执行器再通知 InnoDB，将 redo log 的状态从 "prepare" 改为 "commit" 然后刷盘，完成整个事务的提交。

为什么需要？保证 redo log 和 binlog 两份日志的一致性。如果不做两阶段提交，先写哪个都会 crash 后不一致。

### 8. binlog 刷盘成功但 redo commit 前宕机，MySQL 如何保证一致性？

**Q:** binlog 刷盘成功但 redo commit 前宕机，MySQL 如何保证主从一致？

**A:**

MySQL 重启时触发崩溃恢复，步骤为：
1. 扫描 redo log 找出所有 prepare 状态事务；
2. 按事务 XID 校验磁盘 binlog，若存在对应完整事务，则强制将 redo log 标记为 commit 完成提交；
3. 最终主库数据与已同步 binlog 的从库保持一致。

### 9. 如何使用 EXPLAIN 分析 SQL 执行计划？核心看哪些字段？

**Q:** EXPLAIN 结果怎么看？重点看哪几个字段？

**A:**

在 SELECT 前加 EXPLAIN 即可查看 MySQL 优化器的执行计划。核心关注以下字段：
- **id**：执行顺序，id 越大越先执行。
- **type**：访问类型，性能关键指标。
- **key**：实际使用的索引。
- **rows**：估算扫描行数，越小越好。
- **Extra**：附加信息，判断优化情况。

### 10. type 字段的性能排序？哪些是必须优化的？

**Q:** type 字段不同值性能好坏怎么排序？哪些值必须优化？

**A:**

性能从优到劣排序：`system > const > eq_ref > ref > range > index > ALL`

目标：尽量达到 range 或以上。
红线：避免 ALL（全表扫描）。

---

### 11. Extra 中常见的关键值分别代表什么？

**Q:** EXPLAIN 的 Extra 字段各个值什么意思？

**A:**

- **Using index**：使用了覆盖索引，性能优秀。
- **Using where**：存储引擎返回数据后，Server 层再过滤。
- **Using temporary**：使用临时表，常见于 GROUP BY，性能差。
- **Using filesort**：外部排序，说明 ORDER BY 字段无索引，性能差。
- **Using index condition**：使用了索引下推，优化良好。

### 12. 如何通过 EXPLAIN 判断索引是否生效？

**Q:** 怎么看 EXPLAIN 结果知道索引用上了没有？

**A:**

- 看 **key** 字段：不为 NULL 表示用到了索引。
- 看 **type** 字段：不是 index 或 ALL，表示索引有效。
- 看 **key_len** 字段：长度越长，说明联合索引使用得越充分。

### 13. MySQL 深分页（如 LIMIT 1000000, 10）为什么慢？如何优化？

**Q:** 深分页为什么慢？有哪些优化方法？

**A:**

深分页慢是因为 MySQL 需要扫描并丢弃大量前置记录，offset 越大越慢。

**优化方案：**
1. **延迟关联**：先通过索引仅查询主键 ID，再关联原表查询所需字段，避免回表开销；
2. **书签 / 游标分页**：用上一页最后记录的 ID（或排序字段值）作为条件，直接向后定位，避免 offset 扫描，例如 `WHERE id > 1000000 LIMIT 10`。
