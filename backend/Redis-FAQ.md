## 基础概念篇

### 1. Redis 为什么这么快？

**Q:** Redis 单线程为什么还能这么快？

**A:**

1. **纯内存操作**：数据全部存在内存，读写速度远快于磁盘
2. **IO 多路复用**：用 epoll/kqueue/evport 多路复用模型，单线程高效处理多个连接，避免线程上下文切换开销
3. **单线程设计**：避免了多线程锁竞争开销，CPU 缓存命中率更高
4. **数据结构简单优化**：特殊设计的数据结构（压缩列表、跳表）对内存做了优化，操作更快

<br>

**误区澄清**：单线程指的是**网络请求处理和命令执行是单线程**，并不是 Redis 整个服务只有一个线程。Redis 在持久化、异步删除、集群同步等操作都会开额外线程。

---

### 2. Redis 和 Memcached 区别？怎么选？

**Q:** Redis 和 Memcached 都是缓存，区别是什么？

**A:**

| 对比维度 | Memcached | Redis |
|----------|-----------|-------|
| **数据类型** | 只支持简单字符串 | 支持 String/List/Hash/Set/Zset 多种数据类型 |
| **持久化** | 不支持持久化，重启丢数据 | 支持 RDB/AOF 持久化，可以做数据持久化存储 |
| **线程模型** | 多线程，依赖锁机制 | 单线程（命令处理），无锁竞争 |
| **内存管理** | LRU 自动淘汰 | 多种淘汰策略，支持自定义过期时间 |
| **集群模式** | 支持分布式，但原生不好用 | 原生支持哨兵、集群，高可用方案成熟 |
| **适用场景** | 纯缓存、小数据简单缓存 | 缓存+持久化存储，复杂数据结构操作 |

**选择原则**：
- 简单缓存场景，数据可以丢 → 选 Memcached（多核扩展性好一点）
- 需要持久化、复杂数据结构、高可用 → 选 Redis（现在绝大多数场景都用 Redis）

---

### 3. Redis 有哪些数据类型？各自适用场景？

**Q:** Redis 支持哪几种数据类型？分别用在什么场景？

**A:**

| 数据类型 | 底层实现 | 适用场景 |
|----------|----------|----------|
| **String** | 简单动态字符串 SDS | 缓存对象、计数器、分布式 ID、session 存储 |
| **List** |  quicklist（压缩列表+双端链表） | 消息队列、最新消息列表（比如朋友圈最新动态）、栈/队列 |
| **Hash** | 压缩列表/哈希表 | 存储对象信息（比如用户信息），比多个 String 更省空间 |
| **Set** | 哈希字典 | 共同好友、标签去重、抽奖随机抽取、判断存在 |
| **ZSet** | 跳表 | 排行榜、延时队列、按分数排序、分页查询 |
| **Bitmap** | 字符串二进制位 | 用户签到统计、布隆过滤器、用户在线状态统计 |
| **HyperLogLog** | 基数统计 | 统计 UV 不精确去重，省空间 |
| **Geo** | 有序集合 | 地理位置计算、附近的人 |

### 4. Redis 过期键删除策略是什么？

**Q:** Redis key过期了是立刻删除吗？怎么删除？

**A:**

**三种删除策略：**

1. **定时删除**：设置过期时间的时候，创建定时器，过期了立刻删除。优点：对内存友好；缺点：对 CPU 不友好，过期键多的时候占用 CPU 影响性能
2. **惰性删除**：key 过期不删，等下次访问的时候检查，过期了就删掉。优点：不占 CPU；缺点：对内存不友好，很多过期键没人访问就一直占着
3. **定期删除**：每隔一段时间，随机抽查一批 key，删除过期的。平衡点，实际常用

**Redis 实际用的是：** **惰性删除 + 定期删除** 组合。

即使这样，还是会有过期键没被删掉留在内存里，这时候 **内存淘汰策略** 就会起作用。

---

### 5. Redis 有哪些内存淘汰策略？

**Q:** Redis 内存满了怎么办？哪些淘汰策略？

**A:**

Redis 提供 8 种淘汰策略：

| 策略 | 含义 |
|------|------|
| **volatile-lru** | 从过期键中淘汰 LRU 最近最少使用的 |
| **allkeys-lru** | 从所有键中淘汰 LRU 最近最少使用的 |
| **volatile-lfu** | 从过期键中淘汰 LFU 最不经常使用的 |
| **allkeys-lfu** | 从所有键中淘汰 LFU 最不经常使用的 |
| **volatile-random** | 从过期键中随机淘汰 |
| **allkeys-random** | 从所有键中随机淘汰 |
| **volatile-ttl** | 从过期键中淘汰 ttl 最小的（快过期的） |
| **noeviction** | 不淘汰，写请求直接返回错误 |

**生产推荐**：`allkeys-lru` 或者 `allkeys-lfu` 是现在常用方案。

## 数据结构篇

### 6. 为什么 SDS 比 C 原生字符串好？

**Q:** Redis 为什么自己设计 SDS 不用 C 字符串？

**A:**

C 原生字符串：
- 长度需要用 `strlen` 遍历计算，O(n) 复杂度
- 不能存二进制（因为 `\0` 结尾）
- 修改长度容易缓冲区溢出

SDS 改进：
- **O(1) 获取长度**：结构体保存 len 属性，不用遍历
- **二进制安全**：不依赖 `\0` 结束，可以存任意二进制数据
- **空间预分配**：修改字符串时按需分配多余空间，减少内存分配次数
- **惰性空间释放**：缩短字符串时不立刻回收，留着给后面用

---

### 7. 跳表是什么？为什么 ZSet 用跳表不用红黑树？

**Q:** 有序集合为什么用跳表实现？

**A:**

**跳表核心思想**：在有序链表上增加多层索引，通过空间换时间，把查找复杂度从 O(n) 降到 O(log n)，相当于二分查找的链表实现。

**为什么不用红黑树：**
1. 跳表实现更简单，代码好写好调试
2. 跳表是有序的，范围查询非常方便（顺着往后遍历就行），红黑树范围查询要做中序遍历麻烦一些
3. 跳表插入删除更稳定，平均 O(log n)，不会像红黑树插入删除会频繁调整树结构
4. 跳表更容易做并发访问，调整只影响局部，不需要全树 rebalance

---

### 8. 什么是压缩列表？什么时候用压缩列表？

**Q:** 压缩列表是干什么的？为什么能省内存？

**A:**

压缩列表是 Redis 为了节省内存设计的一种特殊编码的连续内存数据结构。

**特点：**
- 所有元素连续存放在一块内存里，不存指针，省掉指针空间（64位系统一个指针8字节）
- 不同元素不同编码长度，小整数用更少字节存，极端一个整数只要 1 字节
- 缺点：插入删除可能需要连锁更新，因为每个元素的前置长度变了，所以只适合元素少、元素小的场景

**什么时候用：**
- Hash 当字段少、字段值小的时候用压缩列表
- ZSet 当元素少、元素小的时候用压缩列表
- 元素多了或者大了就转成哈希表/跳表

## 持久化篇

### 9. RDB 和 AOF 区别？各自优缺点？怎么选？

**Q:** Redis 两种持久化方式区别？

**A:**

**RDB（Redis Database）**：全量快照，把当前内存中所有数据生成一个二进制快照保存到磁盘。

**AOF（Append Only File）**：增量日志，把所有写命令追加到文件末尾，重启的时候重放一遍命令恢复数据。

| 对比维度 | RDB | AOF |
|----------|-----|-----|
| **文件大小** | 小（二进制压缩） | 大（明文命令，比 RDB 大） |
| **持久化方式** | 全量快照 | 增量追加 |
| **恢复速度** | 快（直接加载二进制） | 慢（要重放所有命令） |
| **数据安全性** | 容易丢数据，最后一次持久化之后的数据丢了 | 更安全，可以配置 fsync 策略，最多丢一秒数据 |
| **文件压缩** | 默认压缩 | 不压缩 |

**优缺点总结：**
- RDB 优点：文件小，恢复快，适合做备份；缺点：不能实时持久化，容易丢数据
- AOF 优点：数据安全，不容易丢；缺点：文件大，恢复慢

**生产怎么用：** 一般**同时开 RDB + AOF**，重启优先用 AOF 恢复数据，更完整；RDB 用来做备份。

---

### 10. RDB 触发机制是什么？

**Q:** 什么时候会触发 RDB 持久化？

**A:**

**两种触发：**

1. **自动触发**：满足 `save m n` 配置（比如 900 秒有至少 1 个键改变就自动 save）
2. **手动触发**：执行 `bgsave` 命令或者 `save` 命令
   - `bgsave`：后台异步执行 RDB，主线程继续处理请求，推荐
   - `save`：主线程直接执行，会阻塞所有请求，不推荐

**注意**：`bgsave` 是 fork 一个子进程来做 RDB，父进程继续处理请求。fork 过程会阻塞主线程，但是生成 RDB 过程不阻塞。

---

### 11. AOF 重写是什么？为什么要重写？

**Q:** AOF 为什么需要重写？怎么重写？

**A:**

**为什么要重写：** AOF 是追加命令，很多命令都是对同一个 key 重复修改，会导致 AOF 文件越来越大，里面很多无用命令（比如多次 increment 最后只需要保留最终值），重写就是把多余命令去掉，只保留最终结果，缩小 AOF 文件体积。

**怎么重写：**
- 后台开启一个子进程，遍历内存中所有数据，把每个 key 最终状态写成一条命令，写入新的 AOF 文件
- 新写入完成之后，原子替换旧文件，完成重写
- 重写过程中主线程继续处理请求，新的写命令同时写到旧 AOF 和重写缓冲区，保证不丢数据

---

### 12. Redis 持久化是主线程做的吗？会阻塞吗？

**Q:** RDB/AOF 会阻塞 Redis 主线程吗？

**A:**

- **RDB bgsave**：fork 子进程的时候会短暂阻塞主线程，生成 RDB 过程子进程做，不阻塞主线程
- **AOF 重写**：和 RDB bgsave 一样，fork 阶段阻塞，重写子进程做，不阻塞
- **AOF 刷盘**：如果是 `appendfsync everysec`，刷盘是后台线程做，不阻塞主线程

只有 fork 那一下会短暂阻塞，其他时候都不阻塞。

## 集群高可用篇

### 13. 什么是缓存雪崩？怎么解决？

**Q:** 缓存雪崩是什么意思？怎么预防？

**A:**

**缓存雪崩：** 大量缓存 key 同时过期，或者缓存宕机了，导致所有请求都打到数据库，数据库扛不住直接挂了。

**解决方法：**
1. **均匀过期时间**：给不同 key 的过期时间加上一个随机值，避免同时过期
2. **缓存高可用**：哨兵或者集群部署，避免单点宕机整个缓存挂了
3. **降级限流**：缓存失效后，用互斥锁或者排队，只允许少量请求过数据库，其他直接返回默认值或者降级
4. **预热热点数据**：提前把热点数据加载进缓存

---

### 14. 什么是缓存穿透？怎么解决？

**Q:** 缓存穿透是什么？怎么解决？

**A:**

**缓存穿透：** 缓存和数据库都没有的数据，请求一直打数据库（比如黑客故意用不存在的 ID 攻击）。

**解决方法：**
1. **缓存空值**：数据库查不到也缓存一个空值，下次请求直接从缓存回，不过要设置短过期
2. **布隆过滤器**：把所有存在的 key 提前放到布隆过滤器，不存在直接拦住，不用去数据库
3. **接口层校验**：比如 ID 范围合法性格验，非法直接拒绝

---

### 15. 什么是缓存击穿？怎么解决？

**Q:** 缓存击穿和雪崩穿透有什么区别？怎么解决？

**A:**

**缓存击穿：** 某个**热点 key** 过期了，刚好这个 key 瞬间大量请求打过来，请求都打到数据库。和雪崩区别是雪崩是大量 key 同时过期，击穿是单个热点 key 过期。

**解决方法：**
1. **互斥锁**：只有一个线程去数据库查，查到重建缓存，其他线程等着，缓存好了再读缓存
2. **永不过期**：热点 key 不设置过期时间，后台异步更新缓存
3. **提前预热**：热点 key 在上线前提前加载进缓存

---

### 16. 主从复制原理是什么？

**Q:** Redis 主从复制怎么工作的？

**A:**

**全量复制 + 增量复制：**

1. **同步阶段**：从节点连接主节点，触发全量复制，主节点生成 RDB 快照发给从节点，从节点加载 RDB
2. **命令传播阶段**：全量同步完成之后，主节点把这期间新的写命令发给从节点，从节点执行，保持一致
3. **后续增量复制**：主节点每接收一个写命令，就同步给所有从节点

**断点续传**：现在 Redis 支持增量复制，如果从节点断开重连，只需要同步断开期间缺少的命令，不用全量重传。

---

### 17. 哨兵机制是什么？哨兵怎么选主？

**Q:** 哨兵解决什么问题？Leader 选举怎么选？

**A:**

**哨兵作用：**
1. **监控**：不断检查主节点和从节点是否存活
2. **自动故障转移**：主节点宕机了，自动从从节点选一个升主，把其他从节点指向新主，通知客户端地址变更
3. **配置中心**：客户端可以问哨兵当前主节点是谁

**选主流程：**
1. 首先过滤掉宕机的从节点
2. 然后选择**优先级最高**的从节点（优先级配置越小越高）
3. 优先级一样选**复制偏移量最大**的（复制的数据最完整最新）
4. 都一样选**runid 最小**的节点

---

### 18. Redis 集群分片原理是什么？

**Q:** Redis 集群怎么分片的？怎么找到 key 在哪个节点？

**A:**

Redis 集群采用 **hash slot（哈希槽）** 分片：

- 整个集群固定分成 **16384 个槽**
- 对 key 用 `CRC16(key) % 16384` 得到槽号，然后映射到对应节点
- 每个节点负责一部分槽，增减节点只需要移动槽就行，不用全部重新分配
- 支持批量操作，只要 key 在同一个 slot 可以 mget 等

**ASK 转向 vs MOVED 转向：**
- MOVED：槽已经稳定归到另一个节点了，告诉客户端永久重定向
- ASK：正在进行槽迁移，当前槽不在这里，让客户端去目标节点问一下，是临时重定向

---

### 19. 什么是 RedLock？为什么需要 RedLock？

**Q:** 分布式锁用 Redis 怎么实现？RedLock 是什么？

**A:**

**单机 Redis 实现分布式锁：**
- `SET lock_key random_value NX PX 30000` （NX 不存在才设置，PX 设置过期时间防止死锁）
- 释放锁用 Lua 脚本：`if redis.call('get',KEYS[1]) == ARGV[1] then return redis.call('del',KEYS[1]) else return 0 end`，判断是自己的锁才删除

**为什么需要 RedLock：** 单机 Redis 做分布式锁，单点宕机锁就没了，有单点问题。RedLock 是多实例 Redis 实现分布式锁，解决单点问题。

**RedLock 算法流程：**
1. 获取当前时间戳
2. 按顺序向 N 个（一般 5 个）独立 Redis 实例请求加锁，每个实例都设置相同锁名和随机值，每个锁都有超时时间，只有超过半数（N/2+1）实例加锁成功，并且总耗时小于锁超时，才算加锁成功
3. 释放锁：给所有实例都发释放锁命令

**争议：** RedLock 理论上很完美，但实际工程中依赖时钟同步，现在很多人认为也没必要，大多数场景单机锁其实够用了，或者用 Zookeeper 做分布式锁更可靠。

## 缓存问题篇

### 20. 怎么解决 Redis 热点 key 大 key 问题？

**Q:** 热点 key 大 key 是什么问题？怎么解决？

**A:**

**热点 key：** 某个 key 访问量特别大，集中打在一个 Redis 节点，把这个节点打垮。

**解决热点 key：**
1. 客户端本地缓存：把热点 key 缓存在应用本地，减少 Redis 访问
2. 拆分热点 key：比如把一个热门商品 key 拆成多个副本，分散到不同节点
3. 限制单机 QPS，熔断降级

**大 key：** 单个 key 存了特别大的数据（比如一个 key 存几 MB 几十 MB），导致：
- 网络传输慢
- 删除的时候阻塞主线程（删除大 key 释放内存要很久）
- 数据倾斜，某个节点内存占用太大

**解决大 key：**
- 拆分：比如一个大 list 拆成多个小 key
- 压缩：大文本压缩存储
- 不要用 DEL 删除大 key，用 `UNLINK` 异步删除，后台慢慢释放不阻塞主线程

### 21. Redis 为什么会产生缓冲区溢出？怎么解决？

**Q:** 复制缓冲区溢出是什么问题？

**A:**

全量复制的时候，主节点生成 RDB 之后，把这期间新的写命令存在复制缓冲区，发给从节点。如果从节点网络慢，或者 RDB 生成太慢，缓冲区满了就会溢出，导致复制失败断开重连，恶性循环。

解决：调大 `client-output-buffer-limit slave` 配置。

---

### 22. Redis 管道 pipeline 有什么用？原理是什么？

**Q:** pipeline 为什么能提速？

**A:**

pipeline 就是把多个命令打包一次性发给 Redis，减少网络往返次数。

- 不使用 pipeline：N 个命令需要 N 次网络往返，每次往返耗时主要是网络延迟
- 使用 pipeline：一次性把 N 个命令发过去，结果一次性回来，只需要 1 次网络往返

**注意：** pipeline 打包的命令不能太多，不然一次发包太大也会慢。

| 考点 | 核心结论 |
|------|---------|
| Redis 为什么快 | 纯内存、IO 多路复用、单线程无锁竞争 |
| 过期删除 | 惰性删除 + 定期删除，过期没删靠内存淘汰 |
| 内存淘汰 | 生产一般用 allkeys-lru / allkeys-lfu |
| RDB vs AOF | RDB 全量快照文件小恢复快，AOF 增量更安全，一般都开 |
| 缓存雪崩 | 大量 key 同时过期 / 缓存宕机 → 均匀过期、高可用、限流降级 |
| 缓存穿透 | 查不到的数据一直打DB → 缓存空值 / 布隆过滤器 |
| 缓存击穿 | 热点 key 过期大量请求打DB → 互斥锁 / 永不过期 |
| 跳表为什么好 | 实现简单、范围查询方便、插入删除稳定 |
| 主从复制 | 全量同步 + 增量命令传播 |
| 哨兵作用 | 监控、自动故障转移、选主 |
| 集群分片 | 16384 个哈希槽，CRC16 取模定位节点 |
