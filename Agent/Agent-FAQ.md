## 基础概念篇

### 1. 什么是 AI Agent？它和普通大模型聊天机器人有什么区别？

**Q:** 什么是智能体 AI Agent？为什么现在 Agent 这么火？

**A:**

**AI Agent（智能体）** 是**能够自主感知环境、自主做出决策、自主执行动作来完成目标的大语言模型应用**。

简单说，普通大模型就是"你问一句它答一句"，而 Agent 能够：
1. **自主规划**：把一个复杂目标拆解成一步步可执行的子任务
2. **调用工具**：需要信息的时候知道去调用外部工具/API 获取
3. **反思纠正**：执行错了能发现错误，自己调整重试
4. **持续迭代**：直到最终完成用户目标才停止

**核心区别：**

| 维度 | 普通聊天机器人 | AI Agent |
|------|----------------|----------|
| **交互模式** | 一轮用户输入 → 一轮模型输出 | 多轮交互：模型 → 工具调用 → 观察 → 模型反复思考 |
| **自主性** | 完全等用户指令 | 自主规划、自主调用工具、自主纠错 |
| **能力边界** | 只能基于模型已有知识 | 能访问外部世界、能执行操作、解决复杂任务 |
| **适用场景** | 问答、闲聊、创作 | 复杂任务自动化、工作流自动化、自主代理 |

Agent 火的原因：大模型本身的推理能力已经足够强了，现在需要让它**真的能落地解决实际复杂问题**，Agent 就是让大模型从"会说"到"会做"的关键架构。

### 2. AI Agent 的核心组成部分有哪些？分别起什么作用？

**Q:** 一个完整的 AI Agent 应该包含哪些模块？

**A:**

主流 AI Agent 的五大核心组件：

```
┌─────────────────────────────────────────────────────────────┐
│                         User Input                           │
└────────────────────────────┬────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  1. Planning（任务规划）                                     │
│  - 把大目标拆解成子任务   │  思考下一步做什么                │
└────────────────────────────┬────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Memory（记忆）                                           │
│  - 存储对话历史、中间结果、用户信息 供后续步骤参考          │
└────────────────────────────┬────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Tool Use（工具调用）                                     │
│  - 调用外部工具获取信息、执行操作                            │
└────────────────────────────┬────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  4. Reflection（反思）                                       │
│  - 回顾执行结果，总结经验，纠正错误                          │
└────────────────────────────┬────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  5. Execution（执行输出）                                    │
│  - 生成最终结果返回给用户                                    │
└─────────────────────────────────────────────────────────────┘
```

**各组件作用：**
1. **Planning**：解决"先做什么后做什么"的问题，复杂任务拆解
2. **Memory**：存储上下文和中间结果，让 Agent 能记住之前做了什么
3. **Tool Use**：让 Agent 能和外部世界交互，拿信息、做操作
4. **Reflection**：让 Agent 能从错误中学习，自我修正
5. **Execution**：执行任务，输出最终结果

### 3. Agent 的 Memory（记忆）分哪几种？怎么设计？

**Q:** Agent 为什么需要记忆？分哪些类型？各存什么内容？

**A:**

Agent 的记忆一般分三级：

| 记忆类型 | 存储内容 | 存储位置 | 作用 |
|----------|----------|----------|------|
| **Sensory Memory / 即时记忆** | 当前轮次用户输入、最新观察结果 | Context 窗口 | 处理当前请求 |
| **Short-term Memory / 工作记忆** | 最近几轮对话历史、中间步骤结果 | Context 窗口（总结后压缩） | 让 Agent 知道任务进度，依赖前面结果 |
| **Long-term Memory / 长期记忆** | 用户偏好、历史任务、知识沉淀、经验总结 | 向量数据库 + 检索 | 跨会话记住用户信息，沉淀经验 |

**工程设计要点：**
- Context 窗口有限，短期记忆不能无限放，需要**滚动总结压缩**，保留关键信息扔掉无关内容
- 长期记忆用向量库存储，需要的时候检索相关内容进 Context
- 用户隐私信息记得加密，做好权限控制

## 主流范式篇

### 4. 什么是 ReAct 框架？ReAct 和 CoT 有什么区别？

**Q:** 介绍一下 ReAct（Reasoning + Acting）框架，它解决了什么问题？

**A:**

**ReAct = Reasoning + Acting**，是 Agent 最经典的框架，把推理和动作交替进行。

**核心思想：**
```
Thought（思考） → Action（动作） → Observation（观察） → Thought（思考）...
```

- **Thought**：大模型用自然语言思考下一步该做什么
- **Action**：调用工具执行动作（查询信息、执行操作）
- **Observation**：拿到工具返回的结果
- 重复这个过程，直到问题解决

**CoT（Chain-of-Thought） vs ReAct 对比：**

|  | CoT | ReAct |
|--|-----|-------|
| **核心** | 只有推理（Thought），没有外部动作 | 推理 + 动作交替，能调用工具 |
| **知识来源** | 只有模型内部知识 | 能从外部获取新知识 |
| **适用场景** | 模型已有知识能解决的推理题（数学、逻辑） | 需要外部信息、需要执行操作的复杂任务 |
| **幻觉问题** | 容易编造事实 | 能基于外部真实信息回答，幻觉少很多 |

一句话：CoT 让模型"想明白了再说"，ReAct 让模型"边想边做，不会就查"。

### 5. 什么是 Reflection（反思）？Agent 为什么需要反思？怎么实现？

**Q:** 反思机制在 Agent 中起什么作用？常见实现方式是什么？

**A:**

**Reflection（反思）** 就是 Agent 执行完一步或完成整个任务后，**自己回顾一下刚才的过程，看看哪里做得不好，怎么改进**。

**为什么需要反思：**
- 大模型第一次尝试经常错，反思能让它自我纠正
- 复杂任务多轮调用后容易偏离目标，反思能拉回正轨
- 从错误中学习，下一次能做得更好

**常见实现方式：**

**1. 自省式反思**（最简单常用）：
```
请你回顾刚才解决问题的过程，分析：
1. 哪些步骤做对了？
2. 哪里出现了错误？为什么错？
3. 下次怎么改进？
```
让大模型自己分析自己，输出改进意见，然后重试一次。

**2. 二元评价反思**：
让大模型给上一步输出打分（0-10分），分数低就重新生成。

**3. 基于反馈的反思**：
拿到用户反馈或者工具错误信息，反思为什么错了，修正参数重试。

**4. 蒙特卡洛树搜索 MCTS 反思**：
尝试多条路径，评估结果，选最好的路径（高级，计算量大）。

**工程实践：** 大多数场景用自省式反思就足够好用了，实现简单，提升明显。

### 6. 什么是 Chain-of-Thought？什么是 Tree-of-Thought？什么是 Graph-of-Thought？区别是什么？

**Q:** 这些不同的"Thought"框架各有什么特点？怎么选？

**A:**

都是针对大模型推理的提示框架，演进路线：

| 框架 | 核心思想 | 优点 | 缺点 | 适用场景 |
|------|----------|------|------|----------|
| **CoT 链式思考** | 一步一步线性推理，链条只有一条 | 简单，实现方便 | 错一步全错，不能回退 | 简单推理问题 |
| **ToT 树式思考** | 把推理过程做成树，每一步探索多个分支，评估后选最优 | 能回溯，能修正，准确率高 | 计算量大，需要多调用几次模型 | 复杂推理，需要探索 |
| **GoT 图式思考** | 推理是一个图，多个思考节点可以互相依赖，不是简单的线性/树状 | 能处理依赖复杂的问题，更灵活 | 复杂度最高 | 复杂多依赖任务，论文/代码生成 |

**趋势：** 从线性 → 树形 → 图，越来越灵活，但计算成本也越来越高。工业界对大多数应用，CoT + ReAct 足够了。

### 7. 什么是 Self-Consistency？它解决了什么问题？

**Q:** Self-Consistency 是什么方法？什么时候用？

**A:**

**Self-Consistency（自一致性）** 是一种简单有效的推理优化方法：

> 让大模型对同一个问题生成多个不同的推理路径，然后投票选出最一致的答案。

**做法：**
1. 设置温度 `temperature > 0`，让模型输出有随机性
2. 同一个问题，让模型生成 k 个不同的答案（通常 k=3~5）
3. 对最终答案进行投票，得票最多的作为最终答案

**解决的问题：**
- 大模型单次推理容易错，多次采样投票能显著提高准确率
- 对数学推理、逻辑题效果特别好

**缺点：** 需要多次调用模型，成本和延迟更高。

## Planning 任务规划篇

### 8. Agent 怎么做任务规划？有哪些常见方法？

**Q:** 用户说"帮我安排一个三天上海出差"，Agent 怎么拆解这个任务？

**A:**

常见任务规划方法分四类：

**1. 零样本规划：ReAct 循环**
   - 不提前做整体规划，走一步看一步
   - 每一步模型思考下一步该做什么
   - 优点：灵活，能根据执行结果动态调整；缺点：容易走偏，短视

**2. 整体规划：Plan-and-Execute**
   - 第一步先让模型把整个任务完整拆解成有序子任务列表
   - 然后按顺序一个个执行
   - 优点：整体观强，不容易漏步骤；缺点：计划赶不上变化，中间出问题难调整

**3. 分层规划**
   - 先做高层计划（大步骤），执行每个大步骤时再做详细分解
   - 兼顾整体观和灵活性，是目前工业界常用方法

**4. DAG 规划**
   - 对于固定业务流程，提前定义好任务依赖的 DAG（有向无环图）
   - 按依赖关系执行，可并行无依赖节点
   - 优点：可控稳定，适合企业固定流程；缺点：不适合开放式任务

**例子（安排出差）：**
```
整体规划分解：
1. 查询用户要求的日期区间航班
2. 选择合适航班预订
3. 查询会议地点附近酒店
4. 预订酒店
5. 把预订信息整理成行程单发给用户
```
然后按顺序一步步执行，就是 Plan-and-Execute。

### 9. 什么是 Function Calling？它和 Agent 是什么关系？

**Q:** Function Calling 是 Agent 必须的吗？

**A:**

关系：**Function Calling 是 Agent 调用工具的手段，Agent 需要 Function Calling，但 Function Calling 不一定需要 Agent**。

- **Function Calling**：是大模型的能力，让模型能输出结构化的函数调用请求
- **Agent**：是一个完整的应用架构，需要规划+记忆+工具调用+反思
- Agent 需要工具调用能力，所以需要 Function Calling（或类似能力）
- 但你也可以单独用 Function Calling，不需要完整 Agent 架构（比如只让模型帮你转成 API 请求）

一句话：Function Calling 是 Agent 工具箱里的"螺丝刀"，是 Agent 的一个核心零件，但不是整台机器。

## Reflection 反思篇

### 10. Reflexion 是什么？和普通 Reflection 有什么区别？

**Q:** 看到论文里提 Reflexion，它是一种什么反思方法？

**A:**

**Reflexion** 是 DeepMind 提出的一种强化学习 + LLM 的反思方法，简单说就是：**让大模型作为"奖励模型"，给自己的输出打分，并生成改进的反思意见**，然后用这个反思来指导下一次尝试。

**和普通 Reflection 的区别：**
- 普通 Reflection：一般只是当前任务做完了回顾一下，帮助完成当前任务
- Reflexion：把反思结果存下来，**跨任务累积经验**，让模型以后遇到类似任务能做得更好，类似强化学习

**核心步骤：**
1. 尝试解决任务
2. 评估结果（可以是大模型评估，也可以是规则/环境反馈）
3. 生成口头反思："刚才哪里错了，下次应该怎么改"
4. 把反思存入经验库
5. 下次遇到类似任务，把相关反思检索出来放进 Prompt，指导决策

效果：在编程、推理任务上能明显提升正确率。

## 多智能体篇

### 11. 什么是多智能体（Multi-Agent）？为什么需要多智能体？单智能体不够吗？

**Q:** 什么时候用多智能体架构？它解决了单智能体什么问题？

**A:**

**多智能体**就是**多个不同角色的 Agent 一起协作完成任务**，每个 Agent 有自己的专长和职责。

**为什么需要多智能体：**
1. **角色扮演分工专业化**：让不同 Agent 扮演不同角色（产品经理 + 工程师 + 测试），比一个 Agent 做所有事质量更高
2. **互相评审减少错误**：一个 Agent 写代码，另一个 review，能提早发现bug
3. **解决复杂任务**：任务太复杂，一个 Agent hold 不住，拆分给多个 Agent 分工
4. **模拟交互场景**：比如对话系统、辩论、游戏，本来就需要多个角色

**典型例子：**
- 研究者 + 评审者：一个生成，一个找错
- 产品 + 研发 + 测试：模拟软件开发流程
- 法官 + 控方 + 辩方：法律辩论场景

### 12. 常见的多智能体协作架构有哪些？

**Q:** 多智能体之间怎么协作？有哪些典型模式？

**A:**

常见四种协作模式：

**1.  Supervisor 模式（主管-工人）**
   - 一个主 Agent（Supervisor）做规划、分任务
   - 分给多个工人 Agent 执行
   - 主 Agent 汇总结果
   - 优点：简单好控；缺点：主 Agent 容易成为瓶颈

**2.  点对点对等模式（Peer-to-Peer）**
   - 所有 Agent 地位平等，谁需要帮助就发消息给对应的 Agent
   - 比如 AgentA 写完代码，直接发给 AgentB 去 review，不需要经过主管
   - 优点：灵活；缺点：协调复杂，可能兜圈子

**3.  辩论模式（Debate）**
   - 多个 Agent 对同一个问题给出不同观点，互相辩论
   - 最后汇总得出更全面结论
   - 适合开放性问题、需要多角度思考的场景

**4.  流水线模式**
   - 任务按流程走：Agent1 做需求分析 → Agent2 写代码 → Agent3 做测试 → Agent4 写文档
   - 适合流程固定，分工清晰的场景

目前工业界用得最多的是 **Supervisor 模式**，简单有效，容易实现。

### 13. 什么是 AutoGPT？它和传统 Agent 有什么区别？为什么没落地普及？

**Q:** AutoGPT 当年很火，为什么现在好像用得不多了？

**A:**

**AutoGPT** 是第一个爆火的开源自主 Agent 项目，核心特点：
- 完全自主：用户只给一个大目标，AutoGPT 自己规划、自己调用工具、自己迭代直到完成
- 长期记忆：用向量数据库存储长期记忆
- 持续迭代：可以自己改进自己的代码

**为什么没能大规模落地：**
1. **容易走偏**：完全自主容易偏离用户目标，"幻觉漫游"，跑半天不知道跑哪去了
2. **成本太高**：一步一调用，完成一个简单任务要调用十几次几十次模型，成本高延迟大
3. **不可控**：出了问题很难调试，用户很难干预
4. **很多场景不需要完全自主**：大多数实际场景人机协作更好，用户在关键环节确认一下体验更好

现在的趋势是**"半自主"Agent**：关键步骤（比如支付、确认预订）需要用户确认，不是完全撒手让 Agent 自己瞎跑。

## 工程实践篇

### 14. Agent 和 RAG 是什么关系？为什么 RAG 系统里经常需要 Agent？

**Q:** RAG 一定要用 Agent 吗？AgentRAG 和普通 RAG 有什么区别？

**A:**

**关系：** RAG 和 Agent 不是替代关系，是互补关系，可以结合使用。

- **RAG**：解决"怎么从知识库找到相关上下文给大模型"
- **Agent**：解决"怎么自主规划一步步完成复杂任务"，可以用 RAG 作为它获取知识的一个工具

**AgentRAG（基于 Agent 的 RAG）** 和普通 RAG 的区别：

| 维度 | 普通 RAG | AgentRAG |
|------|----------|----------|
| **检索方式** | 一次检索，直接生成 | 可以多次检索，边检索边思考，不对就再查一次 |
| **问题理解** | 假设用户问题是清晰的 | 可以拆解复杂问题，一步一步查 |
| **能处理的问题** | 简单问答，问题和知识库直接匹配 | 复杂问题，需要多步推理才能找到答案 |
| **成本** | 低，一次检索一次生成 | 高，可能需要多轮检索多轮推理 |

**什么时候需要 Agent + RAG：**
- 用户问题复杂，需要拆解成多个子问题依次查询
- 需要判断检索结果是否足够回答，不够就再查
- 需要整合多个来源的信息才能回答

如果只是简单知识库问答，普通 RAG 足够；复杂问答场景用 AgentRAG 准确率更高。

### 15. 如果 Agent 选错了工具怎么办？怎么处理？

**Q:** 模型判断错了，选了不对的工具调用，工程上怎么处理？

**A:**

分层处理：

**第一层：提前预防**
- 给每个工具写清晰准确的描述，说明什么时候该用它
- 少样本示范，给几个正确选择工具的例子放 Prompt 里
- 用微调让模型学会更好的工具选择（如果有数据）

**第二层：错误检测**
- 调用工具后拿到结果，让模型判断："刚才选的工具对吗？结果能回答问题吗？"
- 如果不对，让模型重新选一个工具再调用

**第三层： fallback 机制**
- 如果多次选错，或者所有工具都不适用，回答用户："这个问题我没有合适的工具能解决，请换个问题或者补充信息"

**面试要点：** 好的工具描述比复杂算法更重要，描述清晰了，模型选错概率低很多。

### 16. 什么是 Toolformer？它和现在的 Function Calling 有什么区别？

**Q:** Toolformer 是什么模型？

**A:**

**Toolformer** 是 Meta AI 在 2023 年发表的论文，是最早让大模型学会自己调用工具的工作之一。

核心思想：
- 在预训练模型基础上，**微调让模型学会在需要调用工具的时候，自动输出调用指令**
- 用自监督学习方式，不需要大量人工标注数据
- 模型自己决定"什么时候该调用工具""调用哪个工具"

和现在 Function Calling 的区别：

| Toolformer | Function Calling |
|------------|------------------|
| 模型自主决定什么时候调用 | 框架/开发者定义好工具，模型按描述选 |
| 自监督训练出来的能力 | 多是 SFT 教出来，或者闭源模型原生支持 |
| 学术上开创了方向，工业界用得少 | 现在主流方案，工业界广泛用 |

可以说 Toolformer 是先驱工作，现在的 Function Calling 是它思想的工程化落地。

### 17. 工具调用失败了 Agent 应该怎么处理？

**Q:** 工具调用失败（网络错、超时、返回错误），Agent 怎么应对？

**A:**

分情况处理：

1. **参数错误**：返回错误信息给模型，让模型修正参数重试
2. **网络超时/临时错误**：重试几次（退避重试），还不行就换备用工具或者告诉用户服务不可用
3. **工具返回结果不对**：让模型看一下结果是不是答非所问，不对就重新选工具或者重新调用
4. **工具根本不存在**：这是模型幻觉，记录下来，告诉用户工具不存在

工程上一般重试 1~2 次还是错，就转用户澄清或者兜底回答，不要一直重试浪费成本。

### 18. 怎么防止 Agent 无限循环调用工具？有哪些手段？

**Q:** Agent 陷入死循环重复调用同一个工具怎么办？

**A:**

工业界一般三重防护：

1. **最大步数限制**：设置一个绝对上限（比如 15~20 步），超过就停止，总结结果或者转人工
   - 绝大多数任务 20 步以内能完成，超过了说明卡住了

2. **重复动作检测**：
   - 检测到连续两次调用同一个 Skill 并且**参数完全一样**，说明陷入循环了
   - 停止调用，问用户："我卡在这一步了，你能给我点提示吗？"

3. **成本预算控制**：
   - 给每个任务设置 token 预算或者金钱预算
   - 超过预算直接停止，避免成本失控

4. **进度检查**：每几步就让模型自己检查："你现在离目标还有多远？是否还在正确的路上？" 偏了就及时拉回来。

### 19. Agent 调用工具参数错了怎么办？怎么处理？

**Q:** 模型生成的工具参数不对/缺失/格式错了，工程上怎么处理？

**A:**

分层处理，绝大多数问题能解决：

**第一层：提示优化 + 少样本**
- Prompt 里写清楚参数要求，给正确例子，减少出错概率

**第二层：参数校验 + 模型自修复**
- 后端用 JSON Schema 校验参数
- 如果错了，把错误信息返回给模型："你参数错了，错在哪，修正一下重新输出"
- 模型一般能自己修正，成功率 80%+

**第三层：意图澄清**
- 如果是用户没给全信息（比如"取消订单"没说订单号），让模型直接问用户要缺失信息

**第四层：降级兜底**
- 修复几次还是错，就告诉用户："抱歉我没能理解，请换个说法或者手动输入信息"

**工程要点：** 一定要给模型**结构化的错误信息**，它才能修复。只说"参数错了"不如说"`order_id` 字段缺失，请补充"。

### 20. Agent 延迟太高怎么办？怎么优化？

**Q:** 多步调用每次都要等模型推理，用户体验差，怎么优化？

**A:**

几个优化方向：

1. **并行调用**：多个无依赖的工具调用一起发，并行执行，节省时间
   - 比如要查天气+查股价，能一次调用两个工具，不用等一个完了再调另一个

2. **缓存**：相同查询结果缓存，直接返回不用再调
   - 用户问两次同样问题，直接用缓存结果

3. **提前规划 + 预执行**：用户输入一半，模型就能预测下一步要查什么，提前调用工具，用户输完你结果已经回来了

4. **模型选型**：不是每一步都要用 GPT-4o/Claude Opus，简单参数提取、错误修正能用小模型就用小模型，快还便宜

5. **精简 Context**：只把必要的上下文放进去，context 太长会让推理变慢

**典型数据：** 优化得当，大多数简单任务能在 1~2 秒内返回，复杂任务控制在 5 秒以内用户就能接受。

### 21. 多个工具调用之间数据怎么传递？中间结果放哪？

**Q:** 上一步工具返回的结果，下一步要用，存在哪？

**A:**

两种常见方案：

**方案一：放进 Context（推荐大多数场景）**
- 把所有中间结果按照对话历史格式放进当前请求的 Context 里
- 优点：简单，模型能直接看到，不需要额外检索
- 缺点：Context 窗口有限，步骤多了放不下

**方案二：存在外部存储，需要的时候检索**
- 中间结果存在向量库或 key-value 存储
- 需要的时候，模型决定要检索哪个中间结果，取出来放进 Context
- 优点：不占 Context 窗口，能支持非常多步骤
- 缺点：多一次检索，实现复杂一点

**工程实践：** 10 步以内放 Context 就行，简单直接；超过 10 步的复杂任务用外部存储。

### 22. 手撕代码：用 Python 实现一个简单的 ReAct 循环 Agent

**Q:** 实现一个最简版的 ReAct Agent 框架。

**A:**

```python
from typing import List, Dict, Callable, Any
from dataclasses import dataclass

@dataclass
class Tool:
    name: str
    description: str
    func: Callable
    input_schema: dict

class SimpleReActAgent:
    def __init__(self, llm, max_steps: int = 10):
        self.llm = llm  # 大模型调用接口
        self.tools: Dict[str, Tool] = {}
        self.max_steps = max_steps
        self.history: List[dict] = []
    
    def register_tool(self, tool: Tool):
        self.tools[tool.name] = tool
    
    def build_system_prompt(self) -> str:
        tool_descs = "\n".join([
            f"- {t.name}: {t.description}" 
            for t in self.tools.values()
        ])
        
        return f"""你是一个能调用工具帮助用户解决问题的智能助手。

可用工具：
{tool_descs}

工作方式：
你一步步思考：
1. Thought: 你用自然语言思考下一步该做什么
2. Action: 如果你需要调用工具，格式为: Action: tool_name(param1=value1, param2=value2)
3. 你会得到工具返回的 Observation，然后继续思考
4. 当你得到最终答案，输出: Answer: 你的最终回答

开始！
"""
    
    def parse_action(self, output: str) -> tuple[str, dict] | None:
        """解析模型输出，提取 Action"""
        if "Action:" not in output:
            return None  # 是最终 Answer
        
        # 简化解析，生产环境可用正则更健壮
        action_line = [line for line in output.split('\n') if 'Action:' in line][0]
        action_part = action_line.replace('Action:', '').strip()
        
        # 解析工具名和参数（这里简化实现）
        tool_name = action_part.split('(')[0]
        # 生产环境应该用更健壮的参数解析
        return tool_name, {}  # 简化版省略参数解析
    
    def run(self, user_query: str) -> str:
        self.history = []
        self.history.append({"role": "system", "content": self.build_system_prompt()})
        self.history.append({"role": "user", "content": user_query})
        
        for step in range(self.max_steps):
            # 调用大模型
            response = self.llm(self.history)
            self.history.append({"role": "assistant", "content": response})
            
            # 解析是否是 Action
            action = self.parse_action(response)
            if action is None:
                # 最终回答
                if "Answer:" in response:
                    answer = response.split("Answer:")[1].strip()
                else:
                    answer = response
                return answer
            
            tool_name, args = action
            if tool_name not in self.tools:
                observation = f"Error: Tool '{tool_name}' not found."
            else:
                try:
                    tool = self.tools[tool_name]
                    result = tool.func(**args)
                    observation = str(result)
                except Exception as e:
                    observation = f"Error: {str(e)}"
            
            self.history.append({"role": "system", "content": f"Observation: {observation}"})
        
        return f"超出最大步数 {self.max_steps}，未能完成任务。"
```

**关键点：**
- 工具注册：每个工具有名、描述、函数、参数 schema
- ReAct 循环：Thought → Action → Observation 重复
- 最大步数限制防止死循环

## 安全与评估篇

### 23. Agent 可能面临哪些安全风险？怎么防护？

**Q:** Agent 能调用工具执行操作，有什么安全风险需要注意？

**A:**

Agent 因为能执行操作，安全风险比普通聊天机器人更大，主要风险：

| 风险 | 描述 | 防护 |
|------|------|------|
| **越权操作** | Agent 调用了用户没有权限的工具/接口 | 每次工具调用都要做权限校验，放在服务端 |
| **恶意指令注入** | 用户让 Agent 执行删除文件/系统命令 | 禁止给 Agent 系统命令权限，高危操作需要用户二次确认 |
| **Prompt 注入** | 第三方内容（比如检索到的文档里）藏恶意指令，诱导 Agent 做坏事 | 内容分隔，输入净化，权限隔离 |
| **工具滥用** | Agent 反复调用收费 API 导致成本爆炸 | 设置成本预算，速率限制，最大步数限制 |
| **数据泄露** | Agent 把敏感信息返回给了不该看的人 | 输出过滤，敏感信息脱敏，会话级权限控制 |

**核心防护原则：**
1. **高危操作必须人工确认**：删除、支付、修改系统配置这类操作，一定要弹给用户确认了再执行
2. **最小权限原则**：Agent 只给它完成任务必须的权限，不要给超管权限
3. **所有操作可审计**：日志记下 Agent 调用了什么，干了什么，出问题能回溯

### 24. 怎么评估一个 Agent 的好坏？有哪些评估指标？

**Q:** Agent 上线前怎么评估它好不好用？

**A:**

分客观指标和主观指标：

**客观指标：**
- **任务成功率**：100 个任务能成功完成多少个（核心指标）
- **平均步数**：完成任务需要多少步，步数越少越好
- **平均延迟**：完成任务平均需要多少时间
- **成本**：完成任务平均消耗多少 token，平均多少钱

**主观/用户体验指标：**
- **用户满意度**：用户给打分
- **结果正确性**：答案对不对，由人标注判断
- **行为合理性**：思考步骤是否合理，有没有多余步骤

**常用评估方法：**
- **人工评测**：选一批典型测试用例，让人来打分（最靠谱，成本高）
- **大模型作为裁判**：用更强的大模型（比如 GPT-4o）来给结果打分（成本低，一致性尚可）
- **基准数据集**：用公开的 AgentBench 这样的基准测试集来比

## 架构选型篇

### 25. 什么时候用单智能体？什么时候用多智能体？

**Q:** 开发新功能，怎么选单 Agent 还是多 Agent？

**A:**

**选单 Agent 的场景：**
- 任务相对简单，一个角色就能搞定
- 对延迟和成本要求高
- 追求简单，好调试好维护
- 比如：客服问答、简单工具调用、个人助理

**选多 Agent 的场景：**
- 任务复杂，需要不同专长角色分工协作
- 需要评审校验（一个写一个查，减少错误）
- 本身就是多角色交互场景
- 比如：软件开发（产研测分工）、内容创作 + 审核、辩论推理、企业级复杂流程

**一句话总结：** 能单 Agent 搞定就别用多 Agent，简单胜过复杂。多 Agent 适合确实需要分工的复杂场景。

## 发展展望篇

### 27. Agent 目前还有哪些主要挑战没解决？

**Q:** 你认为 Agent 现在技术上最大的痛点是什么？未来方向在哪？

**A:**

目前 Agent 还面临几个大挑战：

1. **上下文窗口限制**：长时间多步骤任务，中间结果放不下，容易丢失信息
   - 方向：更好的记忆压缩 + 长期记忆检索

2. **错误累积**：前面一步错了，后面全错，不能有效回滚
   - 方向：更好的反思机制、规划验证、回滚补偿

3. **成本和延迟**：多步调用多次推理，成本高延迟大
   - 方向：更好的规划、并行调用、小模型加速规划步骤

4. **可调试可观测性差**：Agent 怎么想的哪一步错了，很难排查
   - 方向：更好的 tracing、日志、调试工具

5. **幻觉漫游**：完全自主 Agent 容易偏离目标，跑题了拉不回来
   - 方向：更好的进度检查、人机协作、半自主架构

## 总结：高频考点速记

| 考点 | 核心结论 |
|------|---------|
| 什么是 AI Agent | 能自主感知、规划、调用工具、执行、反思来完成目标的 LLM 应用，区别于一问一答聊天机器人 |
| Agent 核心组件 | Planning（规划）+ Memory（记忆）+ Tool Use（工具调用）+ Reflection（反思）+ Execution |

| ReAct 框架 | Reasoning + Acting 交替：Thought → Action → Observation 循环，最经典最常用 |
| ReAct vs CoT | CoT 只有推理，ReAct 能调用工具拿外部信息 |

| 什么是 Reflection | Agent 自我回顾执行结果，发现错误，自我纠正 |
| 什么是 Self-Consistency | 多次采样推理路径，投票选最一致答案，提高准确率 |
| 任务规划方法 | ReAct 走一步看一步 / Plan-and-Execute 先整体规划再执行 / 分层规划 / DAG |
| Agent vs RAG | 互补关系：RAG 解决检索知识，Agent 解决自主规划，AgentRAG 用于复杂问答 |
| Toolformer | 最早让大模型学会调用工具的先驱工作，现在 Function Calling 是工程落地 |
| 选错工具怎么办 | 清晰工具描述预防 → 结果验证 → 重新选择 → 兜底 |
| 防止无限循环 | 最大步数限制 + 重复动作检测 + 成本预算控制 |
| 多智能体协作模式 | Supervisor 主从分工（最常用）/ P2P 对等 / 流水线 / 辩论 |
| 参数错了怎么办 | 参数校验 → 返回错误给模型自修复 → 问用户澄清 → 降级兜底 |
| 工具调用失败怎么办 | 参数错就修复，临时错就重试，一直错就换备用工具或兜底 |
| 什么时候用多智能体 | 任务复杂需要分工协作，单智能体搞不定的时候；否则优先单智能体 |
| 核心安全原则 | 高危操作必须人确认，最小权限，所有操作可审计 |
