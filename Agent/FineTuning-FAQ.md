## （我的思考）

> 这里用来记录你读完后的个人思考、总结、面试准备笔记

### 1. NF4 和 AWQ 各适合什么场景？微调后部署的工业界标准流程是什么？

**Q:** NF4 和 AWQ 在微调-部署流程中如何分工？完整标准流程是哪几步？

**A:**

- **NF4 (Normalized Float 4-bit)**: 是为**微调**而生的，核心价值是「零校准、低显存、支持 QLoRA 微调」，让个人用户单张 24GB 显存就能微调 70B 模型，是当前大模型微调的事实标准。
- **AWQ (Activation-aware Weight Quantization)**: 是为**推理**而生的，核心价值是「极致精度 + 推理加速」，适合把微调好的模型量化后部署到生产环境，用极小的精度损失换 75% 显存节省和 2~3 倍推理速度提升。

<br>

**工业界标准 4 步流水线**（从微调完到部署）：

1. **QLoRA（NF4）微调**: 用 NF4 4bit 量化基座训练，目的是省显存、能单卡训，输出是 LoRA 小文件（几十 MB）。注意：原始全精度基座一直保留，没被修改！

2. **合并 LoRA → FP16/BF16 全精度模型**: 把 LoRA 增量合并到原始全精度基座，得到一个全新的全精度模型。**这一步最关键**: 不需要从 4bit 反量化，因为从头至尾保留了原始全精度。

3. **AWQ 量化**: 用合并好的全精度模型，通过 AWQ 工具重新量化成 4bit，得到 AWQ 4bit 模型，精度极高，接近 FP16。

4. **vLLM 部署推理**: vLLM 直接加载 AWQ 4bit 模型，速度最快、精度最高、显存最省。

<br>

**核心分工**: 微调用 NF4 (QLoRA) 省显存，部署用 AWQ 提精度提速度，这是当前最优实践。

---

### 2. QLoRA-SFT 和 QLoRA-DPO 有什么区别？

**Q:** 同样是 QLoRA，SFT 和 DPO 训练方式有什么不同？

**A:**

二者均采用 4bit NF4 量化 + 仅训 LoRA 的低显存训练技术，显存成本极低，单卡即可训练大模型，核心差异体现在三方面：

- **数据格式**: QLoRA-SFT 用 `(prompt, answer)` 二元组数据，QLoRA-DPO 用 `(prompt, chosen, rejected)` 三元组数据；

- **损失函数与训练目标**: QLoRA-SFT 用自回归交叉熵损失，核心是让模型模仿正确回答，学习指令到答案的映射关系；QLoRA-DPO 用对比偏好损失，核心是让模型对齐人类偏好，拉大对好答案和坏答案的概率差；

- **训练依赖关系**: QLoRA-SFT 可独立基于预训练模型开展训练；QLoRA-DPO 必须以 SFT 模型为参考锚点，流程上是串行关系：`预训练模型 → QLoRA-SFT → QLoRA-DPO`。

简单来说，QLoRA 是降低训练成本的技术手段，SFT 和 DPO 的核心区别在于数据、目标和依赖关系。

### 3. 微调中的梯度累加是什么？为什么需要它？小 batch 有什么问题？batch 越大显存占用越高的原因是什么？参数更新由什么决定？为什么梯度累加能模拟大 batch 效果？

**Q:** 梯度累加的原理是什么？为什么能解决显存不足问题？

**A:**

分点精简解答如下：

**1. 什么是梯度累加？**

梯度累加是用时间换空间的训练技巧：显存不足以跑大 batch 时，分多次跑小 batch，每次只做前向、反向传播，不立即更新参数，只把梯度累计起来；等累计 N 个小 batch 的梯度后，再统一用总梯度更新一次参数，最终等效于跑了「单次 batch × N」的大 batch。

<br>

**2. 为什么小 batch 不好？**

参数更新依赖一批样本的平均梯度，小 batch 的梯度仅来自极少数样本，无法代表全局数据的真实优化方向，梯度噪声大、训练不稳定，会导致收敛慢、模型效果差，还无法充分利用 GPU 并行算力。

<br>

**3. 为什么 batch 越大，显存占用越高？**

训练时显存的核心开销来自中间激活值和梯度，二者都和 batch size 正相关：

- 中间激活值：前向传播时，模型每一层要为 batch 内每个样本保存中间计算结果，用于反向传播，batch 越大，需要保存的激活值越多，显存线性增长；
- 梯度：反向传播的梯度计算依赖激活值，batch 越大，计算出的梯度越多，显存占用同步上涨；

模型权重、优化器状态的显存是固定的，和 batch size 无关。

<br>

**4. 参数更新由什么决定？**

参数更新由一批样本的平均梯度决定：输入一批样本 → 前向算损失 → 反向算梯度 → 对梯度求平均 → 优化器用平均梯度更新参数。batch size 的本质，就是定义「一次更新用多少样本计算这个平均梯度」。

<br>

**5. 为什么梯度累加能模拟大 batch 效果？**

梯度累加的核心逻辑，就是把多个小 batch 的梯度累计后求平均，最终得到的平均梯度，和直接跑大 batch 的平均梯度完全一致，因此优化器的更新效果和大 batch 几乎无差。

## 基础概念篇

### 1. 什么是微调（FineTuning）？为什么需要微调？预训练和微调区别是什么？

**Q:** 预训练好了为什么还要微调？

**A:**

| 阶段 | 预训练 | 微调 |
|------|--------|------|
| **数据** | 大规模通用语料 | 特定任务小规模数据 |
| **目的** | 学习通用语言知识和世界知识 | 适配特定任务/风格/领域 |
| **学习率** | 小，学习新知识 | 比预训练大一点 |
| **时长** | 几周几个月 | 几小时几天 |

**为什么需要微调：**
1. 让预训练大模型适配你的**特定任务**（比如客服对话、代码生成）
2. 对齐**特定风格**，输出符合你的要求
3. 注入**领域知识**，提升特定领域性能
4. 对齐人类偏好（RLHF/DPO）

---

### 2. 全参数微调 vs 参数高效微调（PEFT）区别是什么？什么时候选哪个？

**Q:** 全量微调好还是 PEFT 好？

**A:**

| 对比 | 全参数微调 | PEFT（参数高效微调） |
|------|------------|-------------------|
| **训练显存** | 大（7B模型需要多张GPU） | 小（7B 单张 24GB 就能训） |
| **参数更新** | 所有参数都更新 | 只更新很少一部分参数（通常 < 1%） |
| **性能** | 数据足够时最好 | 数据不多时不比全参数差多少 |
| **存储成本** | 每个任务存一份完整权重（7B ~ 13GB） | 只存增量（几 MB） |
| **训练成本** | 高 | 低 |

**选择：**
- ✅ **数据足够（> 100k 样本）+ 多张GPU** → 全参数微调性能更好
- ✅ **数据不多 / 显卡不够** → PEFT，成本低够用了
- ✅ **多任务** → PEFT 每个任务只存小增量，节省存储

## PEFT / LoRA 篇

### 3. 什么是 PEFT？PEFT 有哪些常见方法？

**Q:** 参数高效微调 PEFT 是什么意思？有哪些方法？

**A:**

**PEFT（Parameter-Efficient Fine-Tuning）**：只更新模型中极少一部分参数，就能把大模型适配到新任务，大部分参数冻住不更新，**显存需求和存储都大幅减少**。

常见 PEFT 方法：

| 方法 | 核心思想 |
|------|----------|
| **LoRA**（Low-Rank Adaptation） | 在原权重旁边加一个小的低秩矩阵旁路，只训练这个小矩阵 |
| **Adapter** | 在每一层插入小的瓶颈模块，只训练适配器模块，原权重冻住 |
| **Prefix Tuning** | 在输入前面加可训练前缀向量，只训练前缀 |
| **Prompt Tuning** | 只训练输入的 soft prompt，原模型全冻住 |

**现在最流行就是 LoRA**，简单有效，显存省，性能好。

---

### 4. LoRA 原理是什么？为什么能用？

**Q:** 讲讲 LoRA 的核心思想。

**A:**

**LoRA（Low-Rank Adaptation）** 核心思想：**大模型权重更新矩阵也有低秩特性，可以用两个小矩阵乘积近似**。

原权重矩阵 W 是 `d × k`：
```
ΔW = W × A × B
```
- W 原权重，冻住不更新
- A 是 `d × r`，B 是 `r × k`，r 是秩（很小，比如 8/16/64）
- ΔW 就是增量更新，只训练 A 和 B
- 推理的时候可以把 ΔW 合并到 W 里，没有推理延迟

**为什么有效：** 研究发现大模型适应新任务时，权重增量其实秩很低，所以低秩近似就够了，不需要更新全部参数。

**优点：**
- 显存占用小
- 存储小，每个任务只存 A 和 B
- 推理可以合并，不增加延迟
- 切换任务只要换 A/B 就行，快

---

### 5. LoRA 的 rank 选多大比较好？rank 太小会怎么样？

**Q:** LoRA r 选 8 还是 64？

**A:**

经验值：
- **r=8~16**：大多数任务足够，显存最小
- **r=32~64**：数据多，任务差异大，可以选大点
- **r>128**：很少需要，收益递减

**rank 太小：** 模型 capacity 不够，学不好新任务，性能下降
**rank 太大：** 参数变多，显存变多，接近全参数微调，失去 LoRA 意义

**面试要点**：一般任务 r=8 就够了，想效果好点 r=16/32，不要太大。

---

### 6. LoRA 推理需要额外延迟吗？

**Q:** LoRA 部署的时候会不会变慢？

**A:**

**不会**。因为部署时可以把 LoRA 的增量 ΔW 直接合并到原权重 W 里，推理和原来一模一样，没有额外延迟。

如果要支持多任务热切换，可以不合并，动态加上 ΔW，这会有一点点额外计算，但很小，r 很小影响可以忽略。

---

### 7. LoRA 和 Adapter 对比，各有什么优缺点？

**Q:** LoRA 代替 Adapter 了吗？

**A:**

| 对比 | LoRA | Adapter |
|------|------|---------|
| 额外参数 | 很少 | 比 LoRA 多一点（每层都加一个） |
| 推理延迟 | 可合并，无额外延迟 | 不能合并，推理 always 有额外延迟 |
| 实现 | 简单 | 每层都要改，稍复杂 |
| 性能 | 差不多 | 差不多 |

**现在 LoRA 更流行**，因为可以合并无延迟，使用更方便。

---

### 8. Q-LoRA 是什么？和普通 LoRA 有什么区别？

**Q:** Q-LoRA 量化+LoRA？

**A:**

**Q-LoRA = 量化基座 + LoRA 微调**：
- 把基座权重 4bit 量化，冻住
- 只微调小的 LoRA
- 这样 70B 模型单张 24GB 显存就能训练

**和普通 LoRA 区别：**
- 普通 LoRA：基座一般是 FP16，显存需求大
- Q-LoRA：基座 4bit 量化，显存需求极小，性能下降不多
- 优点：让你在单张消费级GPU上就能微调 70B 模型

## 微调方法篇

### 9. SFT 是什么？为什么大模型 RLHF 之前要 SFT？

**Q:** SFT 全写什么？作用是什么？

**A:**

**SFT = Supervised Fine-Tuning（监督微调）**，就是用人工标注的（prompt, answer）样本对模型微调，让模型学会你要的输出格式和风格。

**RLHF 之前一定要 SFT**：
1. 把预训练模型对齐到你要的任务分布
2. 让模型学会按照人类喜欢的格式输出
3. 给 RLHF 提供一个好的初始点，不然 RL 很难收敛

没有 SFT 直接 RLHF，模型容易训飞，很难学好。

---

### 10. 什么是 RLHF？三个阶段是什么？

**Q:** RLHF 怎么训练 ChatGPT？

**A:**

**RLHF = Reinforcement Learning from Human Feedback**，用人类偏好反馈训练大模型，让模型输出更符合人类偏好。

**三个阶段：**
1. **SFT**：人类写好回答样本，监督微调模型，得到 SFT 模型
2. **训练奖励模型**：人类给多个模型输出排序打分，用这个数据训练一个奖励模型，给好回答打高分
3. **PPO 强化学习**：用奖励模型当奖励信号，用 PPO 微调 SFT 模型，优化输出符合奖励模型（也就是人类偏好）

---

### 11. RLHF 现在有什么问题？

**Q:** RLHF 训练不稳定吗？

**A:**

RLHF 主要问题：
1. **训练复杂**：三个阶段，需要人工标注，需要 PPO 训练，不容易稳定
2. **奖励模型崩塌**：奖励模型有时候会错给高分给不好的回答，模型找到"奖励黑客"
3. **成本高**：需要大量人类标注，训练流程复杂
4. **对齐税**：为了对齐人类偏好，模型能力下降（比如写代码能力下降）

## 偏好优化篇

### 12. 什么是 DPO？为什么 DPO 现在越来越火？比 RLHF 好在哪里？

**Q:** DPO 直接优化偏好，不用 PPO？

**A:**

**DPO = Direct Preference Optimization**，直接用人类偏好数据（哪个回答更好）**直接优化策略**，不需要单独训练奖励模型，不需要 PPO。

**比 RLHF 好在哪里：**
1. **训练简单**：只用一阶段，不需要 RM 不需要 PPO，只有 SFT 一样简单
2. **稳定**：不容易训飞
3. **成本低**：标注一样，但训练流程简单，省成本
4. **性能差不多**：很多任务和 RLHF 效果差不多，更容易复现

**现在趋势：** 对齐偏好越来越多人用 DPO 代替 RLHF，因为简单稳定效果差不多。

---

### 13. DPO 原理是什么？为什么不用 PPO 也能训练？

**Q:** DPO 怎么做到直接优化偏好？

**A:**

DPO 核心思想：把偏好优化问题转化为**对数似然对比损失**。

给定一个 prompt，人类偏好回答 `yw` 胜过 `wl`：
```
Loss = -log(sigmoid(β (log p(yw|x) - log p(wl|x))))
```
就是直接拉大偏好回答和非偏好回答的概率差，让模型更倾向生成偏好回答。

从 KL 约束的 RL 问题推导出来，可以证明等价，所以不需要 PPO，直接算对比损失就行。

**非常简洁**，一张卡就能训 70B，推荐用 DPO 对齐偏好。

---

### 14. IPO, KTO, DPO 是什么关系？

**Q:** 这些直接偏好优化方法有什么区别？

**A:**

都是**直接偏好优化**家族，不用 PPO：
- **DPO**：最流行，简单，效果好，现在推荐用这个
- **IPO**：Improved DPO，改进了正则项，更稳定一点
- **KTO**：知道哪个好不知道哪个坏也能训练，适合只有好评没有坏样本的情况

绝大多数情况用 DPO 就够了。

## 工程实践篇

### 15. 微调的时候怎么选择学习率？LoRA 学习率和全参数微调有什么区别？

**Q:** 微调学习率大概选多少？

**A:**

经验值：
- **全参数微调**：学习率要小，1e-5 ~ 5e-5，太大容易训飞忘掉预训练知识
- **LoRA**：学习率可以大一点，1e-4 ~ 5e-4，因为只更新少量参数，不容易忘

**要点：** LoRA 因为大部分参数冻住，所以可以用更大学习率，收敛更快。

---

### 16. 微调的时候梯度累加是什么？为什么需要？

**Q:** batch size 太小显存不够怎么办？

**A:**

**梯度累加（Gradient Accumulation）**：多跑几个 batch 再更一次参数，相当于变大 batch size。

比如你显存只能跑 batch size 2，想要 batch size 8，就：
```
for step in 1..4:
    forward()
    backward()  # 梯度累加，不更新
step 4:
    optimizer.step()  # 累计 4 个batch 梯度一起更新
```

**为什么需要：** 显存不够，买不起大显存卡，用时间换空间，能跑更大 batch。

---

### 17. 梯度检查点（Gradient Checkpointing）是什么？用来解决什么问题？

**Q:** 训练大模型显存不够还有什么方法？

**A:**

**1. 什么是梯度检查点？**

梯度检查点是用时间换空间的显存优化技术：常规训练会保存全部中间层激活值用于反向求导；开启后仅保留关键节点的激活值，丢弃中间层数据，反向传播时按需重算，以此大幅降低显存占用。

<br>

**2. 解决的核心问题**

专门解决 **大模型训练显存不足（OOM）** 的痛点：训练时显存最大开销是中间激活值，模型越大、batch / 序列越长，激活值占用越高；梯度检查点通过 "不存中间值、反向重算"，砍掉 30%~50%+ 的激活显存，让小显存卡也能跑大模型、长序列训练。

<br>

**3. 优缺点**

✅ **优点**：显存节省效果极强，无精度损失，可与梯度累加、QLoRA 等技术叠加使用

❌ **缺点**：反向重算会增加 20%~50% 训练时间，训练速度变慢

<br>

**4. 适用场景**

模型太大显存放不下、长序列训练、消费级 GPU 微调大模型等显存紧张场景，核心原则：**能跑起来比跑得快更重要**。

---

### 18. 微调完 LoRA 怎么合并权重部署？

**Q:** LoRA 训练完怎么用？

**A:**

两种部署方式：

**1. 合并权重（推荐生产环境）：**
```python
# 把 LoRA 增量合并到原权重
merged_weights = original_weights + alpha * lora_delta
# 保存合并后的权重，直接推理，没有额外延迟
```

**2. 不合并，动态加载（推荐多任务切换）：**
- 原权重不动，需要哪个任务加载哪个 LoRA 增量加上去
- 切换任务快，不用保存多份完整权重

---

### 19. 微调时什么时候需要冻结 embeddings？输入输出 embeddings 要不要冻住？

**Q:** 输入输出 embeddings 要不要冻住？

**A:**

embeddings（嵌入层）是模型的词表 - 语义向量映射层，承载基础语言能力，冻结指固定其参数不更新，不影响模型理解文字。

- **数据少**: 冻结 embeddings，减少训练参数，降低过拟合风险，保留模型通用语言能力，训练更稳定。

- **数据多**: 可不冻结，让模型学习任务专属嵌入表示，适配垂直领域术语，提升任务效果。

- **LoRA 微调**: 默认冻结 embeddings 和模型主干，仅训练 LoRA 低秩矩阵，省显存、防过拟合，是行业标准做法。

## 总结速记

| 考点 | 核心结论 |
|------|---------|
| 全微调 vs PEFT | 全微调数据够显卡够就全微调，否则 PEFT |
| LoRA 原理 | 低秩近似 ΔW = W × A × B，只训练 A/B，可合并不增加推理延迟 |
| LoRA rank | 一般 8~16 足够，越大参数越多显存越多 |
| Q-LoRA | 量化基座 + LoRA，单卡微调 70B |
| SFT 作用 | RLHF 第一步，对齐任务，给 RL 好起点 |
| RLHF 三个阶段 | SFT → 奖励模型 → PPO 强化学习 |
| DPO 优势 | 直接优化偏好，不用 RM 不用 PPO，简单稳定成本低，现在流行 |
| 梯度累加 | 时间换空间，显存不够增大 batch |
| 梯度检查点 | 时间换空间，省显存训练更大模型 |
| PEFT 主流 | LoRA 现在最流行，简单有效 |
