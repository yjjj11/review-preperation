## 基础概念篇

### 1. 什么是 MCP（Model Context Protocol）？它解决了什么问题？

**Q:** 什么是 Model Context Protocol？为什么需要这个协议？

**A:**

MCP 全称 **Model Context Protocol**，是 **Anthropic 开源的**一个开放标准协议，用于**标准化应用程序向大语言模型提供上下文的方式**。

**解决的核心问题：**

在 MCP 出现之前，每个 LLM 应用都需要自己定制开发如何连接不同的数据源（比如本地文件、数据库、API 服务、第三方工具等）。每种数据源都需要写一套自定义的集成代码：

- 要连接 Notion，得写 Notion 集成
- 要连接 PostgreSQL，得写 PostgreSQL 集成
- 要连接 Google Drive，又得写 Google Drive 集成

这样做的结果是：**集成成本极高，数据源越多，代码越难维护，新数据源接入慢**。

MCP 的目标就是提供一个**统一开放的协议**，让任何数据源只要实现一次 MCP 协议，就能被任何支持 MCP 的 LLM 应用使用，实现"一次实现，到处使用"。

---

### 2. MCP 的核心理念是什么？

**Q:** MCP 设计的核心思想是什么？

**A:**

MCP 的核心理念可以总结为三点：

1. **标准化上下文提供**：统一规定了 LLM Client 和 Context Server 之间如何通信、如何获取上下文、如何调用工具，不再需要自定义集成代码

2. **去中心化**：每个数据源自己运行一个 MCP Server，不需要把所有数据都集中同步到一个中央数据库，数据保留在原地，需要的时候通过协议获取

3. **可组合**：一个 LLM 应用可以同时连接多个不同的 MCP Server，每个 Server 提供不同类型的上下文/工具，自由组合

类比理解：MCP 就像是 USB-C 接口，统一标准后，任何设备只要符合标准就能插上去用，不需要为每个设备做不同的插头。

---

### 3. MCP 主要解决什么场景的痛点？

**Q:** 什么时候会用到 MCP？

**A:**

MCP 特别适合以下场景：

- **AI Agent 需要连接多种外部数据源/工具**：Agent 需要从不同地方获取数据、调用不同工具，MCP 标准化后接入成本大大降低
- **企业内部需要整合多个异构数据源**：企业有文档在 Notion、代码在 GitHub、数据在数据库，用 MCP 可以快速统一接入给 LLM 使用
- **需要频繁接入新数据源**：产品需要不断支持用户连接新的数据服务，MCP 让社区贡献一次后所有人都能用
- **数据隐私要求高**：数据不需要离开原有系统，MCP 按需拉取上下文，减少数据暴露范围

### 4. 什么是 Skill？Skill 和 MCP 是什么关系？

**Q:** 面试中常说的 Skill（技能插件）是什么？和 MCP 关系是什么？

**A:**

**Skill（技能）** 是封装了特定外部能力的功能模块，可以理解为"大模型可调用的插件"。每个 Skill 具有：

- 明确的功能描述：说明这个 Skill 是做什么用的
- 结构化的输入输出 Schema（JSON Schema）：定义清楚需要哪些参数，每个参数是什么类型
- 可执行的后端逻辑：可以是 Python 函数、REST API、gRPC 服务等

举个例子：
```python
def get_user_order_status(user_id: str, order_id: str) -> dict:
    # 调用内部订单系统查询
    return {"status": "shipped", "tracking_number": "SF123456"}
```

**Skill 和 MCP 的关系：**
- MCP 是**通信协议标准**，定义了模型如何发现 Skill、如何调用 Skill
- Skill 是**具体的功能单元**，运行在 MCP Server 上，通过 MCP 协议暴露给模型调用
- 类比：Skill ≈ USB 设备，MCP ≈ USB 统一接口协议

### 5. MCP 和传统的 RAG 架构有什么区别？

**Q:** MCP 是不是替代 RAG？和 RAG 是什么关系？

**A:**

MCP **不是 RAG 的替代品，而是互补关系**。MCP 解决的是**如何从不同数据源获取上下文**的协议标准问题，RAG 解决的是**如何检索和利用上下文**生成回答的技术问题。

| 对比 | MCP | RAG |
|---------|-----|-----|
| **本质** | 通信协议标准 | 生成架构技术 |
| **解决问题** | 怎么从各种数据源拿到上下文 | 拿到上下文后怎么检索、怎么用 |
| **层级** | 应用层协议，规范数据交换格式 | 生成算法架构 |
| **关系** | MCP 可以给 RAG 提供数据源，RAG 可以基于 MCP 获取的上下文做检索 | RAG 可以用 MCP 统一接入不同数据源 |

举个例子：你要做一个基于企业内部知识库的 RAG 问答系统，你的知识库分散在 Google Drive、Notion、PostgreSQL 三个地方。你可以：

1. 给这三个数据源分别启动 MCP Server（社区已经有实现好的）
2. 你的 RAG 系统通过 MCP 协议统一从这三个数据源获取文档
3. 拿到文档后，照常做分块、Embedding、检索、生成这些 RAG 步骤

MCP 帮你省掉了分别对接三个数据源的自定义代码。

## 核心架构篇

### 6. MCP 的整体架构是什么样的？有哪些核心角色？

**Q:** MCP 架构分为哪几个部分？各部分做什么？

**A:**

MCP 采用**客户端-服务器（Client-Server）架构**，核心角色只有两个：

```
┌─────────────────────────────────────────────────────────────┐
│  LLM Application / Agent（LLM 应用/Agent）                    │
│  ┌─────────────────────────────────────────────────────────┐│
│  │                   MCP Client                              ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────┬───────────────────────────────┘
                              │ MCP Protocol (JSON-RPC over HTTP/SSE/stdio)
                              │
            ┌─────────────────┴─────────────────┐
            │                                   │
┌───────────────────────┐           ┌───────────────────────┐
│    MCP Server 1       │           │    MCP Server 2       │
│  (Notion 数据源)      │           │  (PostgreSQL 数据源)   │
└───────────────────────┘           └───────────────────────┘
```

**核心角色：**

1. **MCP Client**：运行在 LLM 应用/Agent 一侧，负责：
   - 发起请求：向 MCP Server 请求上下文、调用工具
   - 处理响应：把从 Server 拿到的上下文/结果整理好交给 LLM
   - 协议转换：处理 MCP 协议的编解码

2. **MCP Server**：运行在数据源一侧，每个数据源对应一个 Server，负责：
   - 暴露能力：把自己能提供的上下文（列出资源、搜索、获取内容）和工具通过 MCP 协议暴露出去
   - 处理请求：响应 Client 的查询请求，返回对应数据
   - 权限控制：只有有权限的数据才能被访问

**传输方式：**
- HTTP + SSE（Server-Sent Events）：适合网络通信，远程服务
- stdio（标准输入输出）：适合本地进程间通信，本地工具

### 7. MCP 定义了哪些核心原语（操作）？

**Q:** MCP 协议定义了哪些核心操作？

**A:**

MCP 协议基于 JSON-RPC 定义了以下核心原语：

| 原语类别 | 操作 | 作用 |
|---------|------|------|
| **发现能力** | `list_tools` | 列出这个 Server 提供了哪些工具，每个工具的参数 schema |
| | `list_resources` | 列出这个 Server 提供了哪些可访问的资源（文档、数据等） |
| **获取上下文** | `get_resource` | 获取指定资源的具体内容 |
| **搜索上下文** | `search` | 根据关键词搜索相关资源/内容 |
| **调用工具** | `call_tool` | 调用 Server 提供的工具，传入参数，获取结果 |
| **流式输出** | 流式响应 | 支持大内容流式返回 |

这些就是 MCP 的核心操作，所有实现都要遵循这个标准。

---

### 8. 什么是 MCP Resource？什么是 MCP Tool？两者区别是什么？

**Q:** MCP 中 Resource 和 Tool 怎么区分？什么时候用哪个？

**A:**

| 概念 | 定义 | 作用 | 例子 |
|------|------|------|------|
| **Resource（资源）** | 只读的上下文数据，是给 LLM 用来参考的资料 | 提供背景知识、文档内容等"资料" | Notion 页面、Google Drive 文件、数据库表数据 |
| **Tool（工具）** | 可执行的操作，能改变外部状态或者执行动作 | 让 LLM 能执行操作、写数据、调用外部服务 | 创建 Notion 页面、执行 SQL 查询、调用 API 写数据 |

**核心区别：**
- Resource：读数据，给 LLM 当上下文用，**不改变外部状态**
- Tool：执行操作，可能**改变外部状态**，有副作用

## 工作流程篇

### 9. MCP 的一次典型工具调用流程是什么样的？消息格式有什么规范？

**Q:** 假设模型要调用一个 "发送邮件" 的 Skill，MCP 的完整交互流程和消息格式是什么样的？

**A:**

以一个典型同步调用为例，完整四步流程和消息格式：

**步骤1：模型请求 Skill 调用（Model → Tool Server）**

Client 发送 `CallRequest` 格式请求：
```json
{
  "mcp_version": "1.0",
  "type": "call_request",
  "skill_name": "send_email",
  "call_id": "req_12345",
  "context_id": "sess_abc",
  "arguments": {
    "to": "user@example.com",
    "subject": "Your Order Confirmation",
    "body": "Thank you for your purchase!"
  }
}
```

**步骤2：工具服务执行（Tool Server）**
- 解析 `skill_name`，找到对应 Skill 实现
- 验证参数是否符合该 Skill 的 JSON Schema
- 执行业务逻辑（如调用 SMTP 服务）
- 若成功，构造 `CallResponse`；若失败，构造 `CallError`

**步骤3：返回结果（Tool Server → Model）**

成功响应示例：
```json
{
  "type": "call_response",
  "call_id": "req_12345",
  "result": {
    "message_id": "msg_67890",
    "status": "sent"
  }
}
```

错误响应示例：
```json
{
  "type": "call_error",
  "call_id": "req_12345",
  "error_code": "INVALID_EMAIL_FORMAT",
  "message": "The 'to' field is not a valid email address."
}
```

**步骤4：模型继续生成**
- 模型收到结果后，将其作为上下文的一部分，继续生成自然语言回复

**关键设计点：**
- `call_id` 用于匹配请求与响应，支持并发调用
- `context_id` 用于会话状态跟踪
- 所有字段均有严格 Schema 定义，避免解析歧义

### 10. 如果 Skill 执行很慢（比如要跑 10 秒），MCP 怎么处理？

**Q:** 长时间运行的 Skill 调用如何处理？

**A:**

MCP 原生支持**异步调用模式**：

1.  Client 发起调用后，Server 立即返回 `{"type": "call_accepted", "task_id": "task_111"}` 确认接收
2.  Client/模型可以选择：
    - 轮询：通过 `get_task_status` 定期查询状态
    - 回调：提供 webhook，Server 执行完成后主动回调通知
3.  任务完成后，Client 获取最终结果

这在"生成大数据报告""视频处理""批量导出"等长时间运行场景非常实用。

---

### 11. MCP 如何支持多数据源？怎么同时从多个 Server 获取数据？

**Q:** Agent 需要从多个数据源获取数据，MCP 怎么处理？

**A:**

MCP 原生支持多数据源：

1. **一个 Client 可以同时连接多个 MCP Server**，每个 Server 对应一个数据源
2. Client 可以**并行向多个 Server 发起请求**，每个 Server 返回自己的结果
3. Client 把所有结果汇总后统一交给 LLM，由 LLM 整合多个来源的信息
4. 不同数据源之间互相独立，新增一个数据源只需要新增一个 MCP Server 即可，不需要修改现有代码

## 核心组件篇

### 12. MCP Server 需要实现哪些核心功能？

**Q:** 开发一个自定义的 MCP Server 需要做什么？

**A:**

开发一个符合规范的 MCP Server 需要实现：

1. **协议处理**：遵循 MCP 协议规范，处理 JSON-RPC 请求和响应
2. **能力发现**：实现 `list_tools` 和 `list_resources`，暴露自己的能力
3. **资源处理**：如果提供资源，实现 `search` 和 `get_resource`
4. **工具处理**：如果提供工具，实现 `call_tool`，参数校验，执行对应的操作
5. **传输层**：支持一种或多种传输方式（stdio / HTTP + SSE）
6. **认证授权**：处理客户端身份验证，只有授权的 Client 能访问数据

现在官方已经提供了 SDK（TypeScript SDK 和 Python SDK），大部分样板代码都不用自己写，只需要实现业务逻辑（对接具体数据源）即可。

---

### 13. MCP Client 需要实现哪些核心功能？

**Q:** 在自己的 Agent 中接入 MCP 需要做什么？

**A:**

MCP Client 需要实现：

1. **连接管理**：管理到多个 MCP Server 的连接，维护连接状态
2. **协议编解码**：按照 MCP 协议封装请求，解析响应
3. **能力缓存**：缓存各个 Server 提供的工具和资源列表，不用每次请求都重新发现
4. **请求转发**：把 Agent 的请求转发给对应的 Server，汇总结果返回给 Agent
5. **错误处理**：处理 Server 连接失败、请求超时、返回错误等异常情况

官方 SDK 也提供了 Client 的实现，开发者只需要集成到自己的 Agent 中即可。

## 安全机制篇

### 14. MCP 有哪些安全考虑？如何做权限控制？

**Q:** MCP 怎么保证数据安全？谁能访问什么数据怎么控制？

**A:**

MCP 的安全设计遵循以下原则：

1. **权限在 Server 侧控制**：MCP Server 自己管理权限，因为它最了解自己的数据
   - Client 发起请求时带上身份凭证（Token、API Key）
   - Server 验证凭证，只返回该用户有权限访问的数据

2. **数据按需获取**：不需要把数据提前同步到中央存储，数据始终存在原系统，只在查询时返回需要的片段，减少数据暴露

3. **传输安全**：远程 Server 必须使用 HTTPS 加密传输，防止中间人窃听

4. **输入验证**：Server 必须对 Client 传入的参数（特别是工具调用参数）做严格验证，防止注入攻击

5. **沙箱隔离**：本地工具调用建议在沙箱中运行，防止恶意 MCP Server 执行有害代码

### 15. MCP 会面临哪些安全风险？怎么防护？

**Q:** 使用 MCP 可能会遇到什么安全问题？怎么防护？

**A:**

**常见安全风险：**

1. **恶意 MCP Server**：如果用户连接了不受信任的第三方 MCP Server，Server 可能返回恶意内容，或者通过工具调用执行有害操作
   - **防护**：只允许连接信任的 Server，对本地工具调用做沙箱隔离，限制文件系统访问权限

2. **Prompt 注入**：如果 MCP 返回的资源内容中包含恶意 Prompt 注入指令，可能会诱导 LLM 执行意外操作
   - **防护**：对从 MCP 获取的内容做输入净化，用分隔符明确区分用户指令和 MCP 内容，和 RAG 的防护方法类似

3. **越权访问**：Client 没有正确校验权限，获取了不该拿到的数据
   - **防护**：权限控制一定要放在 Server 侧，不能依赖 Client 做权限检查，Server 每次请求都要重新鉴权

---

### 16. 如果 Skill 本身有漏洞（比如 SQL 注入），该怎么防范？

**Q:** Skill 实现存在安全漏洞怎么办？

**A:**

多层防护：

1. **开发规范约束**：强制使用参数化查询，禁止拼接 SQL 字符串
2. **CI/CD 自动化扫描**：集成 SAST（静态应用安全测试）工具，在代码合并前扫描漏洞
3. **最小权限原则**：数据库连接使用只读账号（除非确实需要写操作），限制 Skill 能访问的表
4. **输入校验**：Skill 层面再做一次参数白名单校验，过滤危险字符
5. **沙箱运行**：高危 Skill 在沙箱环境中运行，即使被攻破也不会影响宿主系统

## 应用场景篇

### 17. MCP 在 AI Agent 架构中扮演什么角色？

**Q:** AI Agent 为什么需要 MCP？MCP 给 Agent 带来什么好处？

**A:**

AI Agent 天生需要**连接很多外部工具和数据源**，MCP 完美解决了 Agent 的工具/数据源接入痛点：

- **降低集成成本**：Agent 只需要实现一次 MCP Client，就能使用所有符合 MCP 规范的工具和数据源
- **社区生态复用**：社区已经实现了很多常用数据源的 MCP Server（Notion、Google Drive、PostgreSQL、GitHub 等），拿来就能用
- **用户自定义扩展**：用户可以自己添加自己需要的数据源，不需要修改 Agent 主程序代码
- **架构清晰**：每个数据源服务独立部署、独立扩展，不影响主 Agent 系统

现在的 Agent 框架（比如 Claude Desktop、OpenAI Agents SDK 等）都在逐步支持 MCP，MCP 正在成为 Agent 工具调用的标准协议。

---

### 18. MCP 适合做企业内部知识库问答吗？为什么？

**Q:** 企业内部知识库用 MCP 架构好不好？有什么优势？

**A:**

**非常适合**，主要优势：

1. **数据不用搬家**：企业文档本来就分散在各个系统（SharePoint、Notion、飞书文档等），不需要把所有文档都同步出来建索引，MCP 直接按需拉取，减少数据同步成本和隐私风险

2. **实时性好**：文档更新了立刻就能搜到，不需要等同步索引更新

3. **接入快**：每个系统社区一般都有现成的 MCP Server，启动就能用，快速搭建原型

4. **权限继承**：可以直接复用原系统的权限体系，不用重新做一套权限，用户只能看到他本来就能看到的文档

**局限性：** 如果文档非常多（百万级以上），纯按需检索性能不如预先建好向量索引的传统 RAG，这时候可以混合架构：MCP 负责接入，预先建向量索引加速检索。

## 对比与选型篇

### 19. MCP vs OpenAPI 有什么区别？为什么不直接用 OpenAPI？

**Q:** OpenAPI 也是 API 规范，为什么还要 MCP？

**A:**

| 对比 | MCP | OpenAPI |
|---------|-----|---------|
| **设计目标** | 专门为 LLM 提供上下文/工具调用优化 | 通用 API 设计规范 |
| **核心原语** | 原生内置了 `search`/`get_resource`/`call_tool` 这些 LLM 需要的操作 | 没有预设操作，完全自定义 |
| **发现机制** | 原生支持自动发现工具/资源，LLM 可以自动理解有哪些能力可用 | 需要人工编写描述，自动发现难度大 |
| **流式支持** | 原生支持流式响应，适合大文本返回 | 需要自己扩展 |
| **适用场景** | LLM 获取上下文、调用工具 | 通用 Web 服务 |

简单说：OpenAPI 是通用规范，MCP 是**针对 LLM 上下文获取场景专门优化**的协议，对 LLM 应用来说更开箱即用，少写很多胶水代码。

### 20. MCP vs OAuth 是什么关系？会互相替代吗？

**Q:** MCP 和 OAuth 怎么配合使用？

**A:**

**完全不替代，互补关系：**

- **MCP** 解决的是 "LLM 如何获取上下文和调用工具" 的 **业务协议问题**
- **OAuth** 解决的是 "第三方应用如何获得用户授权访问数据" 的 **授权问题**

实际使用中，MCP 可以用 OAuth 做授权：
- 用户要连接自己的 Notion 账号，通过 OAuth 授权给 MCP Server
- MCP Server 拿到 Token 后，就能代表用户访问 Notion API
- 之后 MCP 协议正常处理查询请求

### 21. MCP 和 OpenAI Function Calling 有什么区别？

**Q:** MCP 和 OpenAI 的 Function Calling 是一回事吗？核心区别在哪？

**A:**

两者不是一个层面的东西，定位和扩展性区别很大：

| 维度 | OpenAI Function Calling | MCP |
|---------|--------------------------|-----|
| **定位** | 封闭生态内的 API 功能 | 开放、跨厂商的通用协议 |
| **发现机制** | 需要手动传入 tools 列表给模型 | 支持动态发现（`list_tools`），Client 自动获取 |
| **传输格式** | 仅支持 JSON | 支持 JSON、Protobuf、自定义二进制 |
| **状态管理** | 无原生支持 | 原生支持会话上下文（context_id） |
| **错误处理** | 只有简单 error message | 结构化错误码（如 `PERMISSION_DENIED`、`INVALID_ARGUMENT`），便于模型决策 |
| **生态** | 绑定 OpenAI 生态 | 开源开放，任何厂商任何模型都能用 |

简单说：**Function Calling 是"功能"，MCP 是"操作系统级协议"**。Function Calling 解决的是模型如何生成工具调用参数的问题，MCP 解决的是工具如何标准化暴露给任意模型的问题。可以一起用：模型用 Function Calling 生成调用参数，然后通过 MCP 协议调用远端工具。

### 22. Skill 和 Function Calling 中的 Tool 有什么区别？

**Q:** Skill 和 Function Calling 里的 Tool 不是一回事吗？区别在哪里？

**A:**

**Skill** 是从**概念层面**，**Tool** 是**Function Calling 中对可调用功能的称呼，两者其实说的是同一个东西——都是"大模型可调用的外部功能"。但在具体语境和架构中，侧重点有所不同：

| 对比维度 | Skill（MCP生态中的Skill） | Tool（Function Calling 中的 Tool） |
|---------|-----------------------------|--------------------------------------|
| **所属层面** | 强调**封装好的、可独立部署的功能单元 | 强调**被模型调用的功能 | 同一个东西，叫法不同 |
| **架构定位** | 通常运行在**独立 MCP Server 中，可以被任意模型调用 | 通常定义在**模型调用请求**中，由对应一个具体的函数 |
| **生态特征** | 一次实现，**多处复用**，可以被任何支持 MCP 的模型调用 | 每个模型厂商**各自定义格式，需要适配 |
| **部署方式** | 可以独立部署、独立升级 | 通常和业务逻辑绑定在一起 |

更准确的关系：**Skill 是对 Tool 的一种标准化封装方式。在 MCP 生态中，所有通过 MCP 暴露的 Tool 都叫 Skill。

简单说：**Skill 是标准化后的 Tool。

### 23. MCP 对比传统的向量库+RAG 架构，优缺点是什么？

**Q:** 直接用 MCP 还是用传统 RAG？怎么选？

**A:**

| 架构 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **传统 RAG（向量库提前索引）** | 检索快，百万级文档性能好 | 需要数据同步，占用额外存储，实时性差 | 文档量大，更新不频繁，查询性能要求高 |
| **纯 MCP 架构** | 数据不用同步，实时性好，权限继承原系统 | 海量文档检索性能差，每次查询都要远程调用 | 文档分散在多个现有系统，对实时性要求高，文档量不算特别大 |
| **混合架构（推荐）** | MCP 接入 + 提前建向量索引，兼具两者优点 | 工程复杂度稍高 | 大多数企业生产场景 |

**混合架构做法：**
1. 通过 MCP 接入各个数据源
2. 定期（或增量）从 MCP 获取文档，在向量库建索引
3. 查询时先用向量索引检索，再用 MCP 获取最新完整内容
4. 兼顾检索性能和数据实时性

### 24. 什么是 Function Calling？它解决了什么问题？

**Q:** 大模型的 Function Calling 是什么？为什么需要这个能力？

**A:**

**Function Calling（函数调用）** 是大语言模型的一项能力，让模型能够**生成结构化的工具调用请求**，而不只是生成自然文本。

简单说，就是模型知道："这个问题我回答不了，需要调用外部工具"，然后按照约定的格式输出要调用哪个函数、传什么参数。

**解决的问题：**
1. **大模型知识截止日期问题**：模型不知道实时信息（比如今天天气、当前股价），需要调用外部 API 获取
2. **需要执行操作**：模型不能直接帮你发邮件、订酒店，需要调用工具执行实际操作
3. **复杂计算**：大模型计算能力不如计算器，需要调用计算工具

举个例子：
- 用户问："今天北京天气怎么样？"
- 模型知道自己不知道实时天气，于是输出：
  ```json
  {"name": "get_weather", "arguments": {"city": "北京"}}
  ```
- 开发框架调用这个函数拿到天气，再把结果给模型，模型最终用自然语言回答用户

---

### 25. Function Calling 的完整工作流程是什么？

**Q:** 从用户提问到得到回答，Function Calling 的完整步骤是怎样的？

**A:**

典型的 Function Calling 工作流程分为四步：

```
用户提问 → 模型生成函数调用 → 后端执行函数 → 把结果返回模型 → 模型生成最终回答
```

详细步骤：

1. **系统提示 + 函数定义**：开发者把所有可用函数的描述、参数 Schema 告诉模型
   ```
   你是一个助手，可以调用以下工具：
   1. get_weather: 获取城市天气，参数 {"city": string}
   ...
   ```

2. **用户提问**：用户输入问题

3. **模型决策**：模型判断是否需要调用工具
   - 如果不需要工具（比如常识问题）→ 直接回答
   - 如果需要 → 输出结构化的函数调用请求（函数名 + 参数）

4. **后端执行**：开发者根据模型输出，调用实际的函数/API，得到执行结果

5. **结果回填**：把函数执行结果放回对话上下文，再次调用模型

6. **最终回答**：模型基于函数返回结果，生成自然语言回答给用户

**并行调用**：现代大模型（如 GPT-4o、Claude 3.5）支持一次输出多个函数调用请求，后端并行执行后一起返回结果，减少交互轮次。

### 26. 大模型是怎么学会 Function Calling 的？需要微调吗？

**Q:** Function Calling 能力是怎么来的？开源模型需要微调才能支持吗？

**A:**

主流大模型的 Function Calling 能力来源分两种情况：

**闭源模型（GPT-4、Claude 等）：**
- 模型在训练阶段（预训练+SFT）就已经学习了函数调用的格式和能力
- 开发者只需要按照要求提供函数描述和 Schema，模型原生就能生成正确调用
- **不需要额外微调**，开箱即用

**开源模型（Llama 3、Qwen 等）：**
- 基础模型本身没有这个能力，需要**经过 SFT（监督微调）**才会
- 社区已经有很多微调好的版本（比如 Llama 3 Instruct 本身就支持）
- 如果要自己微调，需要准备函数调用对话样本数据

**关键技术点：**
- 关键是用**指令跟随数据**训练，让模型学会理解函数描述、生成符合 JSON Schema 格式的调用
- 现在很多开源基座模型在 SFT 阶段就已经包含了工具调用样本，所以开箱就能用
- 领域特殊函数一般也不需要再微调，靠模型的零样本泛化能力就能搞定

### 27. 如果大模型生成的函数参数不对/缺失怎么办？有哪些应对方案？

**Q:** 模型生成了错误的参数类型，或者必填参数缺失，怎么处理？

**A:**

这是 Function Calling 非常常见的问题，工业界一般多层防护：

**第一层：提示工程优化**
- 给模型清晰的函数描述和参数说明，描述越清楚，出错越少
- 在 System Prompt 中强调："必须严格按照 JSON Schema 输出参数，必填参数不能缺失"
-  Few-Shot：给几个正确调用的例子作为参考

**第二层：调用后参数校验**
- 后端收到模型生成的调用后，先用 JSON Schema 验证参数
- 如果参数不对，不直接执行，把错误信息返回给模型：
  ```
  你刚才的调用参数错误：缺少必填字段 'city'，请修正后重新输出
  ```
- 模型一般能自我修正，重新输出正确的参数

**第三层：意图澄清（面向用户）**
- 如果确实是用户没给全信息（用户说"取消那个订单"没说订单号），返回让模型向用户澄清询问
  ```
  参数错误：order_id 缺失，请询问用户要取消哪个订单
  ```

**第四层： fallback 机制**
- 如果多次重试还是不对，降级转人工或者回答用户"暂时无法完成这个操作"

**工程实践**：第二层重试（把错误返回让模型自修复）成本最低，解决绝大多数问题。

### 28. Function Calling 有哪些常见的坑或挑战？

**Q:** 使用 Function Calling 过程中遇到过哪些问题？怎么解决？

**A:**

常见坑和解决方案：

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| **幻觉参数** | 模型编造不存在的参数值 | 参数校验 + 模型自修复 + 必要时问用户 |
| **格式错误** | 模型输出不是标准 JSON，有多余文字 | 提示强调格式 + JSON 解析容错（提取其中的 JSON 部分） |
| **过度调用** | 明明不需要工具，模型非要调用 | 少样本示范 + 调大温度惩罚 |
| **循环调用** | 模型反复调用同一个函数得不到结果 | 最大步数限制 + 重复动作检测 |
| **权限问题** | 模型调用了用户没有权限的函数 | 调用前后端做权限校验 |
| **错误处理** | 函数执行错误模型不知道 | 把错误信息结构化返回给模型，让它处理 |

**最常见的坑就是格式错误**，实际工程中一定要做好 JSON 解析容错，比如用正则提取出 ```json ... ``` 块再解析。

## 开发实践篇

### 29. 如何设计一个高质量的 Skill？有哪些最佳实践？

**Q:** 假设你要为电商客服设计一个"取消订单"的 Skill，你会怎么设计？

**A:**

我会从**功能设计、安全设计、可观测性**三个维度设计：

**1. 功能设计**
- **单一职责**：明确边界，Skill 只做"取消订单"，不把"退款""重发"等功能混进来
- **清晰的输入 Schema**：明确定义每个参数的类型、描述、是否必填：
```json
{
  "type": "object",
  "properties": {
    "user_id": {"type": "string", "description": "用户唯一ID"},
    "order_id": {"type": "string", "description": "订单ID"},
    "reason": {"type": "string", "enum": ["no_longer_needed", "wrong_item", "other"]}
  },
  "required": ["user_id", "order_id", "reason"]
}
```
- **清晰的输出 Schema**：明确定义返回结果结构

**2. 安全设计**
- **权限校验**：验证用户是否有权操作该订单
- **幂等性保证**：同一订单多次取消应返回相同结果，不重复执行
- **敏感信息过滤**：日志不打印完整订单详情和用户隐私信息

**3. 可观测性**
- **调用日志**：记录 `skill=cancel_order, user_id=xxx, latency=120ms`
- **指标上报**：QPS、成功率、P99 延迟、错误类型分布
- **分布式追踪**：集成 OpenTelemetry，关联到用户会话

### 30. Skill 调用时用户参数不全/模糊指代（如"取消昨天那个订单"）怎么办？

**Q:** 用户没有给全必要参数，Skill 该怎么处理？

**A:**

这种情况属于**意图澄清**，不应该让 Skill 自己处理，应该由**模型+对话管理**来解决：

1. Skill 只需要定义清楚它需要哪些参数，参数不全时返回 `MISSING_ARGUMENT` 错误
2. 模型收到错误后，先调用其他 Skill 获取缺失信息：比如先调用 `list_recent_orders(user_id)` 获取最近订单列表
3. 然后向用户澄清确认："您是要取消订单 #ORD123（金额 ¥299）吗？"
4. 用户确认后，拿到完整参数再调用 Skill

**原则：** Skill 不应该承担模糊输入解析的工作，那是模型的责任，Skill 只需要处理参数完整清晰的请求。

### 31. Skill 调用失败时，模型该怎么处理？有哪些错误恢复策略？

**Q:** 比如 Skill 返回"邮箱格式错误"，模型应该如何响应？

**A:**

我会分三层处理：

**第一层：错误分类**，基于 MCP 结构化错误码做不同处理：
- `INVALID_ARGUMENT`：用户输入问题（如邮箱格式错）→ 引导用户修正
- `PERMISSION_DENIED`：权限不足 → 提示联系管理员
- `INTERNAL_ERROR`：服务故障 → 降级或重试
- `SERVICE_UNAVAILABLE`：服务不可用 → 走备用流程

**第二层：自然语言转化**，不应该直接返回错误码给用户，要转化为用户友好语言：
- ❌ 错误写法："Error: INVALID_EMAIL_FORMAT"
- ✅ 正确写法："抱歉，您提供的邮箱地址格式不正确，请检查后重新提供。"

**第三层：自动恢复（高级）**：
- **参数修正**：对简单错误可以自动修复（比如补全 `@gmail.com`）
- **替代方案**：如果邮件不可用，询问用户"是否改用微信通知？"
- **回退流程**：记录失败，转人工客服

工业界实践：在客服 Agent 中，对 `INVALID_ARGUMENT` 类错误，我们会提取错误字段（如 `to`），生成澄清话术，在下一轮对话自动填充用户新输入到原参数位置。

### 32. 如果 Skill 一直失败（比如网络超时）怎么办？

**Q:** 连续失败怎么处理？

**A:**

需要**熔断机制**：
- 设置阈值：比如连续 3 次失败 → 标记该 Skill 为"暂时不可用"
- 后续请求直接走备用降级流程，不继续调用，避免雪崩
- 同时触发告警通知运维人员排查
- 定期探测恢复，服务恢复后自动打开熔断

### 33. 多个 Skill 怎么协同完成一个复杂任务？比如"帮我安排上海出差"？

**Q:** 用户一个请求需要多个 Skill 配合，怎么处理？

**A:**

这属于**任务规划（Task Planning）**问题，工业界常用 **ReAct（Reason + Act）框架**：

**步骤：**

1. **分解子任务**：模型分析用户意图，拆解成有序子任务
   - 用户需求："帮我安排上海出差，5月1日到3日"
   - 拆解：查询航班 → 预订航班 → 查询酒店 → 预订酒店

2. **循环调用 Skill**：Thought → Action → Observation 循环
   ```
   Thought: 首先需要获取上海五一期间的航班信息。
   Action: call search_flights(origin="Beijing", dest="Shanghai", date_from="2026-05-01", date_to="2026-05-03")
   Observation: [{"flight_id": "CA123", "price": 800, ...}]

   Thought: 选择 CA123 并预订。
   Action: call book_flight(flight_id="CA123", user_id="U123")
   Observation: {"booking_id": "BK789", "status": "confirmed"}

   Thought: 接下来预订机场附近酒店。
   Action: call search_hotels(location="Shanghai Pudong Airport", check_in="2026-05-01", check_out="2026-05-03")
   ...
   ```

3. **异常处理与回滚**：如果订酒店失败，需要取消已订航班，这要求 Skill 提供补偿接口（比如 `cancel_booking`）

**关键技术点：**
- 短期记忆：将中间结果（如航班 ID）存入上下文，供后续步骤使用
- 依赖关系：用 DAG 表示任务依赖，必须等前置任务完成才能执行后续
- 人工确认：关键步骤（如支付）需要用户确认后再执行

### 34. 怎么防止模型无限循环调用 Skill？

**Q:** 模型陷入循环重复调用同一个 Skill 怎么办？

**A:**

一般设置三重防护：

1. **最大步数限制**：设定一个上限（比如最多 10-15 步），超过就停止，总结结果或转人工
2. **重复动作检测**：检测到连续两次调用同一个 Skill 且参数没有变化，就停止并询问用户是否还要继续
3. **成本预算控制**：对付费 API 调用设定成本上限，超过预算就停止

### 35. Skill 调用会增加延迟吗？怎么优化？

**Q:** 每次调用 Skill 都要走网络，会不会让响应变慢？怎么优化？

**A:**

确实会引入额外延迟，但可以通过分层优化控制在可接受范围：

1. **缓存（Cache）**：对幂等 Skill（如 `get_weather`、`get_product_info`），缓存结果 5 分钟，key = `skill_name + hash(args)`，相同请求直接返回缓存
2. **并行调用**：多个无依赖的 Skill 可以并行执行（用 `asyncio.gather` 等），减少总延迟
3. **批处理（Batching）**：多个用户同时查询相同数据，合并为单次 API 调用
4. **预加载（Prefetch）**：在用户输入时，根据关键词预测可能用到的 Skill 提前预热连接
5. **本地缓存热点数据**：常用不常变的数据缓存到 Client 侧

实测参考数据（普通服务器）：
| 场景 | 平均延迟 |
|------|----------|
| 纯文本生成 | ~400ms |
| 调用 1 个 Skill | ~650ms (+250ms) |
| 调用 3 个并行 Skill | ~800ms (+400ms) |

只要 P99 延迟控制在 1.5s 以内，用户体验就比较流畅。

### 36. 怎么监控 Skill 系统的健康状态？

**Q:** 上线后怎么监控？需要关注哪些指标？

**A:**

建立三类监控：

1. **基础性能指标**（Prometheus + Grafana）：
   - QPS：每秒调用量
   - 成功率：成功调用占比
   - P99/P50 延迟：尾部延迟关注 P99 更重要

2. **业务指标**：
   - 调用转化率：多少调用成功完成了业务目标
   - 错误类型分布：哪种错误最多，方便针对性优化

3. **异常告警**：
   - 成功率突然下降 → 自动告警
   - P99 延迟突然飙升 → 自动告警
   - 某个 Skill 连续失败触发熔断 → 自动告警

### 37. 手撕代码：用 Python 实现一个简易支持 MCP 的 Skill 调度器

**Q:** 实现一个简化版的 SkillManager，支持注册和调用。

**A:**

以下是核心实现：

```python
import json
from typing import Dict, Callable, Any
from dataclasses import dataclass

@dataclass
class Skill:
    name: str
    func: Callable
    input_schema: dict  # JSON Schema
    requires_approval: bool = False

class SkillManager:
    def __init__(self):
        self.skills: Dict[str, Skill] = {}

    def register_skill(self, skill: Skill):
        """注册一个 Skill"""
        self.skills[skill.name] = skill

    def validate_input(self, schema: dict, args: dict) -> bool:
        """简易参数校验（生产环境用 jsonschema 库）"""
        for key in schema.get("required", []):
            if key not in args:
                return False
        return True

    def call_skill(self, skill_name: str, arguments: dict) -> dict:
        """调用 Skill，返回 MCP 格式结果"""
        if skill_name not in self.skills:
            return {
                "type": "call_error",
                "error_code": "SKILL_NOT_FOUND",
                "message": f"Skill '{skill_name}' not registered."
            }

        skill = self.skills[skill_name]

        # 参数校验
        if not self.validate_input(skill.input_schema, arguments):
            return {
                "type": "call_error",
                "error_code": "INVALID_ARGUMENT",
                "message": "Input does not match schema."
            }

        try:
            # 执行 Skill
            result = skill.func(**arguments)
            return {
                "type": "call_response",
                "result": result
            }
        except Exception as e:
            return {
                "type": "call_error",
                "error_code": "INTERNAL_ERROR",
                "message": str(e)
            }

# 示例使用
def send_email(to: str, subject: str, body: str) -> dict:
    if "@" not in to:
        raise ValueError("Invalid email format")
    return {"status": "sent", "to": to}

manager = SkillManager()
manager.register_skill(Skill(
    name="send_email",
    func=send_email,
    input_schema={
        "type": "object",
        "properties": {
            "to": {"type": "string"},
            "subject": {"type": "string"},
            "body": {"type": "string"}
        },
        "required": ["to", "subject", "body"]
    }
))

# 调用测试
resp = manager.call_skill("send_email", {
    "to": "test@example.com",
    "subject": "Hi",
    "body": "Hello"
})
print(json.dumps(resp, indent=2))
```

**关键点：**
- Skill 封装了函数和输入 Schema
- `call_skill` 返回标准 MCP 格式
- 包含参数校验和错误处理

### 38. Skill 为什么需要幂等性？怎么实现幂等性？

**Q:** 什么是幂等性？为什么 Skill 需要幂等性？

**A:**

**幂等性**：同一个 Skill 被调用多次，产生的副作用和调用一次结果一样，不会重复创建资源、重复扣款。

**为什么需要：**
- 网络不稳定时，Client 可能重试调用，如果不幂等，会产生重复操作（比如重复发送邮件、重复扣款）
- MCP 支持异步调用、重试，幂等性保证系统一致性
- 用户重复点击也不会产生负面影响

**常见实现方式：**
1. **业务唯一幂等键**：每个调用分配唯一 `request-id`，数据库去重，已经处理过的直接返回结果
2. **乐观锁**：用版本号控制，更新时检查版本，不匹配不执行
3. **前置判断**：执行前先检查状态，比如订单已经取消了就不重复取消
4. **数据库唯一约束**：利用数据库唯一索引防止重复插入

**面试要点：** 对外暴露的 Skill 都应该尽可能实现幂等性，这是工程质量的重要保证。

### 39. Skill 调用失败需要回滚吗？怎么实现回滚？

**Q:** 如果多个 Skill 调用中有一个失败了，需要回滚已经执行成功的 Skill 吗？

**A:**

这取决于业务场景：

**需要回滚的场景：**
- 多个 Skill 组成一个原子业务操作（比如先扣钱再开票，扣钱成功开票失败需要回滚退款）
- 需要保证数据一致性的分布式事务场景

**不需要回滚的场景：**
- 各个 Skill 操作独立，失败不影响前面已经成功的操作
- 查询类只读 Skill，本来就没有副作用，不需要回滚

**常见回滚实现方式：**
1. **补偿式回滚**：为每个 Skill 提供对应的补偿接口（`book_flight` 对应 `cancel_booking`），失败时逆序调用补偿接口
2. **两阶段提交**：先预执行所有 Skill，都成功再提交，任一失败都回滚
3. **保存快照**：执行前保存状态，失败后恢复快照

**工程实践：** 大多数 Agent 场景不需要强一致性，补偿式回滚最常用，实现简单也足够好用。

### 40. 怎么对 Skill 做权限控制？

**Q:** 不是所有用户都能调用所有 Skill，权限怎么设计？

**A:**

多层权限控制方案：

1. **Skill 注册层面**：对 Skill 分级标注（公开/内部/管理）
2. **用户-角色权限绑定**：基于 RBAC（Role-Based Access Control）：
   - 普通用户只能调用业务 Skill，不能调用管理运维类 Skill
   - 管理员可以调用所有 Skill
3. **运行时校验**：每次调用 Skill 都要校验当前用户是否有权限
   - 权限校验放在 MCP Server 侧（不要放在 Client 侧，Client 侧可被绕过）
   - 传入用户凭证，Server 侧验证
4. **细粒度权限**：对资源级权限，比如用户只能取消自己创建的订单，需要在 Skill 内部做业务权限校验

**核心原则：** 权限校验一定要放在 Server 侧，不能信任 Client 侧的校验。

### 41. Skill 版本怎么管理？如何兼容旧版本？

**Q:** Skill 更新了，怎么处理兼容性问题？

**A:**

**常见版本管理策略：**

1. **独立部署不同版本**：`skill-name-v1` 和 `skill-name-v2` 同时注册共存，客户端按需选择
2. **版本号路径**：URL 路径携带版本 `/v1/send_email` `/v2/send_email`
3. **向后兼容**：非 breaking change 直接更新原 Skill，breaking change 出新版本
4. **灰度升级**：先切一部分流量到新版本，验证没问题再全量切

**兼容最佳实践：**
- 新增参数可选，不影响老调用
- 老字段保留，逐步淘汰
- 文档清晰标注版本变化

### 42. 什么是 Skill 编排？有哪些常见编排方式？

**Q:** 多个 Skill 怎么组织起来完成一个复杂任务？

**A:**

**Skill 编排**就是定义多个 Skill 之间的调用顺序、依赖关系、错误处理，让它们协作完成复杂任务。

**常见编排方式：**

| 编排方式 | 特点 | 适用场景 |
|---------|------|----------|
| **ReAct 循环** | 模型一步一步思考 → 调用 → 观察，动态决定下一步 | 未知流程、需要模型决策的开放式任务，Agent 场景最常用 |
| **DAG 有向无环图** | 提前定义好依赖关系，可并行执行无依赖的节点 | 流程固定、依赖清晰的业务流程 |
| **工作流引擎** | 可视化定义流程，支持分支、循环、人工审批 | 企业级固定业务流程 |
| **LLM 动态规划** | 大模型自动分解任务、规划顺序 | 复杂多变的开放式任务 |

**工业界趋势：** 开放式 Agent 场景多用 ReAct 动态编排，固定业务流程多用 DAG/工作流。

### 43. 在项目中快速接入 MCP 需要哪些步骤？

**Q:** 我想在自己的 Agent 项目中接入 MCP，具体怎么做？

**A:**

典型接入步骤：

1. **选择官方 SDK**：根据你的技术栈选择（官方提供 TypeScript 和 Python SDK）
2. **实现 MCP Client**：使用 SDK 快速创建 Client，配置要连接的 MCP Server 列表
3. **发现能力**：Client 启动时连接各个 Server，调用 `list_tools` 获取所有可用工具，注册到 Agent 的工具列表
4. **集成到 Agent 流程**：当 Agent 需要调用工具或获取上下文时，通过 MCP Client 转发请求给对应 Server
5. **处理结果**：Client 返回结果后，整理成 Agent 需要的格式交给 LLM

整个过程如果用官方 SDK，核心代码只需要几十行就能搞定，主要工作是配置要连接的 Server。

### 44. 开发一个自定义 MCP Server 需要哪些步骤？

**Q:** 我有一个自定义数据源，想做个 MCP Server 分享给别人用，怎么做？

**A:**

步骤：

1. **初始化项目**：用官方 SDK 初始化项目（`@modelcontextprotocol/sdk` 或 `mcp` Python包）
2. **定义能力**：定义你的 Server 提供哪些资源，提供哪些工具，每个工具的参数 JSON Schema
3. **实现处理函数**：

5. **测试验证**：用官方提供的 MCP 测试工具测试你的 Server 是否符合规范
6. **部署发布**：可以发布成 npm/pip 包，或者部署成服务

官方 SDK 封装了大部分协议细节，开发者只需要写业务逻辑，非常快就能做出来。

### 46. MCP 使用过程中常见坑有哪些？怎么避免？

**Q:** 使用 MCP 时要注意什么？

**A:**

常见坑和避免方法：

1. **网络延迟问题**：每次查询都要跨网络调用 MCP Server，比本地索引查询延迟高
   - **避免**：对热点数据做缓存，混合架构用预建索引加速检索

2. **错误处理不好**：某个 MCP Server 挂了导致整个 Agent 不可用
   - **避免**：做熔断降级，单个 Server 故障不影响其他 Server，返回兜底提示

3. **权限控制缺失**：把权限检查放在 Client 侧，攻击者可以绕过
   - **避免**：权限一定要放在 Server 侧检查，每次请求都鉴权

4. **参数校验不严格**：工具调用参数不校验，导致注入攻击
   - **避免**：严格按照 JSON Schema 校验所有输入参数，过滤危险操作

**Q:** MCP 现在还是比较新，你觉得它能成为事实标准吗？未来前景如何？

## 总结：高频考点速记

| 考点 | 核心结论 |
|------|---------|
| 什么是 MCP | Anthropic 开源的标准化上下文/工具调用协议，一次实现到处使用 |
| 什么是 Skill | 封装特定外部能力的功能模块，大模型可调用的插件，通过 MCP 暴露 |
| Skill vs Function Calling Tool | Skill 是标准化后的 Tool，MCP 生态中独立部署可复用；Tool 是 Function Calling 中对可调用功能的通用称呼 |
| 解决什么问题 | 工具/数据源生态碎片化，每个模型都要重新适配，接入成本太高 |
| 核心架构 | MCP Client（在 Agent 侧）+ 多个 MCP Server（每个数据源/工具集一个） |
| MCP vs OpenAI Function Calling | Function Calling 是功能，MCP 是跨厂商开放协议；Function 解决模型生成参数，MCP 解决工具标准化暴露 |
| Resource vs Tool | Resource 只读上下文，给 LLM 参考；Tool 是可执行操作，可能改变外部状态 |
| MCP vs RAG | 不替代，互补关系：MCP 解决接入，RAG 解决检索 |
| MCP vs OpenAPI | MCP 专门为 LLM 优化，原生支持能力发现，开箱即用 |
| Skill 设计最佳实践 | 单一职责，清晰 JSON Schema，权限校验，幂等性，可观测 |
| 错误处理 | 分类错误码 → 转用户友好语言 → 自动恢复；连续失败触发熔断 |
| 防止无限循环 | 最大步数限制 + 重复动作检测 + 成本预算控制 |
| Skill 延迟优化 | 缓存热点 + 并行调用 + 预加载 |
| 权限控制 | 权限一定放在 Server 侧，Client 侧不可信；多层防御：审核→RBAC→沙箱→审计 |
| 架构选型 | 推荐混合架构：MCP 接入 + 预建向量索引，兼顾实时性和性能 |
| 优势 | 降低集成成本，数据不用搬家，权限继承，可组合 |
| 劣势 | 海量文档检索性能不如预索引 RAG，有额外网络延迟 |
