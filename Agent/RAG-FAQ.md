## 基础概念篇

### 1. 什么是 RAG？

**Q:** 什么是 RAG（Retrieval-Augmented Generation，检索增强生成）？

**A:** RAG 全称 **Retrieval-Augmented Generation**，是一种结合**外部知识检索**和**大语言模型生成**的混合架构。

**核心原理：**
1. 用户提问后，先从外部知识库（企业私有文档、专业知识库、文档库等）中检索出与问题语义最相关的若干文档片段
2. 将检索到的相关文档作为上下文参考信息，和用户原始问题一起拼接到 Prompt 中喂给 LLM
3. LLM 基于检索到的真实外部信息来生成回答，这样保证答案的准确性，并且每一个结论都可以追溯到来源文档

为什么要这么设计？大语言模型虽然天生生成能力强，但原生存在三个问题：知识截止（训练完就不更新了）、无法访问企业私有数据、容易编造事实产生幻觉。RAG 就是在不需要重新训练/微调模型的情况下，让大模型能够先"查资料"再回答，完美解决了这三个问题。

---

### 2. RAG 的核心优势/好处是什么？为什么需要 RAG？

**Q:** RAG 有什么好处？现在 LLM 已经很强了，为什么还需要 RAG？

**A:** RAG 主要解决了大语言模型原生存在的三个核心问题，带来五大优势：

1. **降低幻觉（Hallucination Reduction）**
   - 问题：大模型天生容易"一本正经地胡说八道"，生成不存在的事实和错误知识
   - RAG 解法：强制要求模型必须基于检索到的内容回答，每一个结论都可以追溯到源文档，大大减少胡说八道，提高可信度

2. **节省训练/微调成本**
   - 问题：如果用全量微调把新知识灌进模型，需要大量数据和计算资源，成本非常高
   - RAG 解法：不需要让模型"死记硬背"海量数据，只需要维护知识库做索引，更新维护成本低很多

3. **知识更新实时性好**
   - 问题：业务知识随时在变，微调跟不上变化速度
   - RAG 解法：知识库更新后只需要重新做索引就能让模型获取新知识，不用重新训练模型，适合知识快速迭代的场景

4. **数据安全可控**
   - 问题：企业私有数据不能随便给出去做微调，有数据泄露风险
   - RAG 解法：不需要把全部私有数据拿去给第三方微调，只在查询时取出相关片段给 LLM，数据暴露范围小，更安全可控

5. **可解释性好，方便溯源**
   - 问题：用户需要知道答案从哪来的，方便验证正确性
   - RAG 解法：答案的每一部分都能找到对应的来源文档，用户可以点击去原文验证，更容易获得用户信任

即便现在大模型（如 GPT-4o）能力已经很强，依然存在三个本质问题：**事实一致性差（还是会幻觉）、上下文窗口有限（不可能塞下整个知识库）、隐性偏见难更新**。RAG 是目前工业界最实用的"给大模型补脑"方案，落地成本比全量微调低很多，收益明显。

### 3. RAG vs SFT（监督微调）有何区别？

**Q:** RAG 和监督微调（Supervised Fine-Tuning，SFT）哪个好？具体区别在哪？什么时候用哪个？

**A:**

| 对比维度 | RAG | SFT（监督微调） |
|---------|-----|----------------|
| **知识存储位置** | 知识存在外部知识库，模型只是"查资料" | 知识被内化到模型权重中，模型靠记忆生成 |
| **知识更新方式** | 更新知识库 + 重新索引即可，成本低，速度快 | 需要重新微调整个模型，成本高，速度慢 |
| **核心优势** | 更新快、灵活、可解释性强、数据安全性好 | 推理更自然、符合模型原生输出分布、不需要额外检索步骤 |
| **核心劣势** | 需要额外检索步骤，延迟更高，工程复杂 | 更新慢、成本高、数据隐私风险高、不可解释 |
| **适用场景** | 知识更新频繁、需要可追溯性、涉及私有数据场景 | 通用能力对齐、写作风格适配、任务迁移场景 |

一句话总结：**SFT 是在模型内部"灌知识"，RAG 是让模型"查资料"。一个靠记忆背下来，一个靠外部分检索。**

**选型建议：** 如果你的知识需要频繁更新，对可追溯性要求高，优先选 RAG；如果你要让模型学习特定的写作风格或者完成特定任务，知识相对固定，选 SFT。实际生产中经常两者结合使用。

### 4. RAG 有哪些局限性/缺点？

**Q:** RAG 有什么缺点？什么问题解决不了？

**A:**

1. **最终效果严重依赖检索质量**
   - 如果检索出来的内容本身就不对、不相关，那后续 LLM 也不可能答对，就是俗话说的 garbage in garbage out（垃圾进垃圾出）
   - 检索质量受分块策略好坏、Embedding 模型质量、向量库召回率等很多因素影响，哪一步出错都会影响最终答案

2. **上下文融合困难**
   - 拼接多个来自不同文档的检索片段时，容易丢失原始语义边界，不同内容之间可能产生冲突
   - LLM 上下文窗口长度有限，塞入太多不相关内容会把真正重要信息稀释掉，导致 LLM 抓不住重点回答错误

3. **生成速度比直接生成慢**
   - 多了向量检索这一步骤，整体延迟比 LLM 直接回答更高，如果用 RAG-Fusion 这种进阶方案延迟会更高

4. **工程复杂度比纯 LLM 问答更高**
   - 需要维护知识库、Embedding 服务、向量数据库、LLM 服务，整个技术链路更长，运维和调试都更复杂

5. **Token 成本更高**
   - 每个用户请求都要把多个检索块塞进 LLM 上下文窗口，Token 消耗比直接问答更大，所以成本也更高

### 5. 什么情况下企业会选择传统搜索而不是 RAG？

**Q:** 为什么有些公司不用 RAG，还坚持用传统搜索？RAG 不是更先进吗？

**A:** RAG 不是银弹，不是解决所有问题的万能药，传统搜索依然有它适合的场景：

1. **性能成本原因**
   - RAG 需要做向量检索 + LLM 生成两道工序，**延迟更高，推理成本更高**
   - 传统搜索只需要做关键词倒排索引搜索，速度快很多，成本也低很多

2. **任务类型原因**
   - 如果用户的真实需求就是**找到哪个文档包含关键词，文档在哪里**，而不是想要一个直接整合好的答案
   - 这种情况下传统搜索直接返回文档列表就足够了，RAG 多出来一层 LLM 生成反而多余，画蛇添足
   - 举个例子：企业内部知识库导航、文档管理系统，用户就是要自己找到文档打开看，传统搜索就够了

3. **可审计性和风险原因**
   - RAG 会让 LLM 重新整合检索到的内容，可能会曲解原文意思，万一回答错了责任不好界定，出问题不好追溯
   - 传统搜索直接返回原文给用户，用户自己判断，内容就是原文一字不改，风险更低，可审计性更好

所以，对于简单的文档查找需求，传统搜索比 RAG 更实用。

### 6. RAG 和传统搜索引擎的核心区别是什么？

**Q:** RAG 比传统搜索好在哪里？本质区别是什么？

**A:**

| 对比维度 | 传统搜索引擎 | RAG |
|---------|-------------|-----|
| **匹配方式** | 关键词/词汇匹配（BM25 等算法），只看字面上匹配不匹配 | 语义向量匹配，真正理解问题的语义，找语义相关的内容 |
| **返回结果** | 返回相关文档列表，需要用户自己打开文档找答案 | 直接综合多个检索片段，整合出一个完整回答直接给用户 |
| **用户体验** | 用户需要自己从返回结果中打开一篇篇文档阅读筛选，找到自己要的信息 | 直接整合好答案给你，用户不需要自己读文档 |
| **适用场景** | 公开网络全网搜索，用户找资料 | 私有知识库、企业内部文档问答，垂直领域问答 |

举个实际例子说明区别：你在公司内部知识库问"公司的请假流程是什么？"
- 传统搜索：返回给你好几篇包含"请假"关键词员工手册文档链接，告诉你我找到这些了，你自己点开找吧
- RAG：直接告诉你第一步找谁审批，第二步走什么系统，假期怎么算，整理好回答，还告诉你这个答案来自哪篇员工手册第几页

### 7. RAG 完整工作流程分为哪几个阶段？

**Q:** RAG 的完整工作流程是怎样的？每个阶段做什么？

**A:** RAG 完整工程流程按顺序分为以下阶段：

| 阶段 | 离线/在线 | 核心工作 |
|------|----------|----------|
| 1. 数据准备 | 离线 | 原始文档提取、清洗、预处理 |
| 2. 文档分块 | 离线 | 将大文档切分成适合检索和上下文的小块 |
| 3. 索引构建 | 离线 | 对每个分块生成 Embedding 向量，存入向量数据库 |
| 4. 在线检索 | 在线 | 用户提问后，将问题向量化，检索出相关文档 |
| 5. 结果排序 | 在线 | 对检索结果排序，选出最相关的 top-K |
| 6. 上下文处理 | 在线 | 对检索结果压缩、拼接，构建最终 Prompt |
| 7. LLM 生成回答 | 在线 | LLM 基于上下文生成最终回答 |
| 8. 效果评估 | 离线/在线 | 评估系统效果，持续优化 |

### 8. RAG 常见应用场景有哪些？

**Q:** RAG 实际用在什么地方？适合哪些业务场景？

**A:** RAG 特别适合**需要基于特定领域私有知识库问答**的场景，典型应用：

- **智能客服机器人**：基于产品文档、帮助中心自动回答用户问题，减少人工客服工作量
- **开发/运维助手**：基于代码文档、运维手册、监控告警文档辅助开发人员快速排查问题
- **医疗辅助诊疗**：基于医学文献、病历知识库辅助医生回答专业问题
- **法律咨询服务**：基于法律条文、判例文书给用户提供法律咨询参考
- **教育学习辅导**：基于教材课件给学生提供个性化答疑辅导
- **企业内部助手**：基于公司内部文档、规章制度回答员工日常问题
- **投资研究/审计**：基于大量研报、财报快速回答分析师研究问题，辅助决策
- **销售支持**：基于产品手册、客户案例帮助销售快速回答客户问题

只要企业有大量私有文档，需要基于这些文档问答，就适合用 RAG.

## 数据准备阶段

**本阶段核心：原始数据 → 干净可用的文本**

### 9. 复杂 PDF 提取困难，表格乱掉、页眉页脚干扰，怎么解决？

**Q:** 复杂 PDF 提取文本效果差，怎么处理？

**A:** PDF 本身布局复杂，纯文本提取会丢失很多结构信息：

**解决方案：**
- 使用布局识别模型（比如微软 LayoutLM 系列）：能识别出文本块、表格、标题、页眉页脚等不同区域，区分不同内容类型，知道哪块是正文哪块是没用的
- 结合 OCR：对于扫描版 PDF 或者图片版 PDF，文字本身就是图片，需要先用 OCR 识别出文字
- 坐标级切分：按照 PDF 页面上的坐标位置切分，保持原文阅读顺序不乱
- 表格做特殊处理：提取的时候把表格结构保留下来，转成 markdown 表格或者 JSON 格式，方便后续处理，不要把表格拆成纯文本丢掉结构

---

### 10. RAG 对 SQL/表格这类结构化数据支持不好，怎么解决？

**Q:** 结构化数据查询难题怎么处理？

**A:** RAG 原生天生擅长处理非结构化文本，对结构化数据（SQL表、Excel表格）需要用混合架构：

**解决方案：**
- 采用 Text-to-SQL + RAG 混合架构：用户问问题，先让 LLM 理解问题，生成正确的 SQL 查询语句，然后去数据库执行 SQL 查询拿到结果数据，再把结果数据交给 LLM 生成自然语言回答
- 表格分块做特殊处理：对于已经提取出来的表格，按行/列语义分块，保留表格结构信息，不要纯文本化扔掉结构
- 结合 Text-to-SQL 技术优化：用专门微调过的 Text-to-SQL 模型提升 LLM 生成正确 SQL 的能力

---

### 11. 非结构化数据处理太慢，大规模知识库数据处理能力跟不上，怎么优化性能？

**Q:** 非结构化文档处理性能瓶颈怎么优化？

**A:**

**优化方向：**
- 批量向量化：利用 GPU 批处理加速，一次批处理多个文档的 Embedding 生成，比一条一条处理快很多
- 流式索引：边处理边构建索引，不需要等全部文档都处理完再一起建索引，加快处理速度
- 分布式检索：多机并行水平扩展检索能力，应对百万级以上大规模知识库场景

---

### 12. RAG 索引输入数据层面怎么优化？有哪些优化方法？

**Q:** 原始数据质量怎么优化？

**A:** 原始数据质量直接影响检索效果，这几步优化必不可少：

- **统一数据格式：** 不同来源（PDF、Word、网页）文档清洗完统一格式再进索引，避免格式不统一带来的噪音
- **冗余去除：** 去掉重复内容、广告、导航、页眉页脚、水印等所有无用信息，只保留有用正文
- **文本归一化：** 统一大小写、统一特殊符号处理、统一字符编码，减少不必要的噪音干扰

## 文档分块阶段

**本阶段核心：大文档 → 合适大小的语义块**

### 13. 常见的文档分块（Chunking）策略有哪些？各自优缺点是什么？怎么选择？

**Q:** 文档分块都有哪些方法？各自适用场景是什么？

**A:** 分块策略是 RAG 最影响效果的第一步，常见分块方法：

| 分块方法 | 做法 | 优点 | 缺点 | 适用场景 |
|---------|------|------|------|----------|
| **固定长度分块** | 按固定 token 数切分，比如每 500 token 切一块 | 简单易实现，速度快 | 可能把完整语义从中间切开，破坏语义完整性 | 快速原型验证，文档结构规整的场景 |
| **按语义分块** | 基于语义相似度切分，语义连贯的放在一块 | 保留完整语义，效果好 | 需要语义模型判断，速度稍慢 | 生产环境，追求效果的场景 |
| **按文档结构分块** | 按照标题、段落、章节天然结构切分 | 尊重文档原有结构，语义完整 | 需要文档本身有清晰结构 | markdown、HTML 等结构化文档 |
| **递归切分** | 先用大结构切，如果太大再递归切小 | 保留文档层级结构，比较灵活 | 参数需要调优 | 大部分通用场景，LangChain 默认推荐 |
| **父-子分块** | 大块保存原文，小块做检索，大块给上下文（ParentDocumentRetriever） | 小块检索准，大块上下文全 | 存储稍大 | 需要完整上下文的场景，效果较好 |

**优化选型建议：** 优先试递归语义分块，效果不好再尝试父-子分块等进阶方法。块大小通常在 200-1000 token 之间，根据你的文档类型调整。

## 索引构建阶段

**本阶段核心：文本块 → 向量索引**

### 14. RAG 索引构建层面怎么优化？有哪些优化方法？

**Q:** 索引构建过程有哪些优化点？

**A:**

- **优化向量生成质量：** 使用 instruction-tuned 微调过的 Embedding 模型（比如 BAAI BGE 系列、阿里巴巴 GTE 系列、OpenAI text-embedding-3 系列等），相比通用非微调模型，检索效果提升非常明显
- **支持增量索引更新：** 不需要每次更新都全量重建索引，只索引新增修改过的文档，提升更新速度
- **索引阶段去重：** 提前去掉重复文档，减少存储占用，减少检索噪音

## 检索计算阶段

**本阶段核心：用户问题 → 召回相关候选文档**

### 15. BM25 是什么？为什么 RAG 常把 BM25 和向量检索结合？

**Q:** BM25 算法是什么？为什么常和向量检索一起用？

**A:**

**BM25 是什么：** BM25 是基于关键词的概率检索算法，是传统关键词检索的主流算法，它会根据关键词在文档中出现的频率（TF）和逆文档频率（IDF）计算文档和查询的相关度得分.

**为什么要和向量检索结合：**
- BM25 擅长**精确关键词匹配**，能抓住必须包含某个关键词的需求，但不理解语义，抓不住同义词
- 向量检索擅长**语义匹配**，能抓住同义词、同义表达，但可能丢了精准关键词匹配
- 两者正好互补，结合起来可以提高召回覆盖率，所以 RAG 中常融合 BM25 + 向量检索结果

---

### 16. Cosine 相似度和 Dot Product（点积）计算相似度有什么区别？该怎么选？

**Q:** 向量检索中，余弦相似度和点积计算相似度有什么区别？实际该怎么选？

**A:**

**公式对比：**
- **Cosine 相似度：** `cos(θ) = (a·b) / (||a|| * ||b||)` —— 先两个向量做点积，然后再除以两个向量的 L2 范数乘积，相当于做了归一化
- **Dot Product（点积）：** 直接就是 `a·b = Σ a_i * b_i` —— 就是原始点积，不做归一化

**核心区别：**

| 区别点 | Cosine 相似度 | Dot Product（点积） |
|--------|--------------|-------------------|
| **是否受向量模长影响** | 完全不受模长（向量长度）影响，只关注方向 | 受模长影响，模长越大的向量得分越高 |
| **归一化处理** | 自动做归一化，消除向量长度的影响 | 不做归一化，保留原始长度信息 |
| **几何意义** | 衡量两个向量的夹角大小，只看方向相似度 | 既看方向相似度，又考虑向量本身的模长大小 |

**选型建议：**

- **用 Cosine 相似度场景：**
  - 如果向量本身已经做过 L2 归一化，那么 cosine 相似度和点积计算出来结果是完全一样的
  - 如果你不希望文档向量的长度影响相似度得分，就用 cosine
  - 大多数通用检索场景，cosine 更常用

- **用 Dot Product 场景：**
  - 如果你的 Embedding 模型训练的时候就是用点积损失训练出来的（比如 OpenAI text-embedding-3 系列官方明确推荐用点积）
  - 如果你想让更重要、出现频率更高的向量天然获得更高分数，就用点积
  - 计算速度略快一点点，因为少一次除法运算

**面试考点总结：** OpenAI 新一代 Embedding 模型（text-embedding-3 系列）官方推荐使用点积而不是 cosine 相似度，因为模型训练方式就是这么设计的，点积效果更好。

---

### 17. 内容缺失问题 —— 知识库明明有就是检索不到，常见原因是什么？怎么解决？怎么优化？

**Q:** RAG 出现内容缺失，检索不到相关内容，怎么解决？

**A:**

**常见原因：**
1. 切片（分块）策略不合理：切得太大或者太小都不好，太大语义太杂抓不住重点，太小信息不完整
2. 向量召回率低：用了质量差的 Embedding 模型，或者向量索引构建不对，导致召回率上不去
3. 知识覆盖不全：知识库本身就真的没有这个内容，那肯定检索不到

**解决思路/优化方法：**
- 调整分段长度：根据你的文档类型和内容结构，实验试出最优块大小（通常 200-1000 token 之间比较常见）
- 使用多向量检索：不只对整块编码，可以对句子、段落不同粒度多级编码，提高召回率
- 增加索引质量评估：离线建完索后用你的常见问题集评估召回率，提前发现问题
- 结合关键词检索：用 BM25 关键词检索做召回，弥补纯向量检索的召回不足，融合两种结果

---

### 18. 为什么会出现相关文档排名靠后没被选中？怎么解决？怎么优化？

**Q:** 相关文档在检索结果中排名太靠后，没进入上下文，导致回答错，怎么解决？

**A:** 这本质就是**召回排序算法问题**：

**解决方案/优化方法：**
1. 优化向量距离计算方式：比如从 cosine 相似度改成 dot-product（点积），有些 Embedding 模型训练方式决定了点积效果更好（比如 OpenAI text-embedding-3 就推荐点积）
2. 引入重排序（rerank）模型：先用向量检索召回更多候选（比如召回 top 50），再用 cross-encoder 重排序模型把真正相关的文档顶到前面，选 top K 进上下文
3. 融合关键词检索结果：把 BM25 关键词检索得分和向量检索得分加权融合，提高排序准确性

---

### 19. 回答不全面，漏了关键信息，是什么原因？怎么解决？怎么优化？

**Q:** RAG 回答不全面漏信息，怎么办？

**A:** 回答不全面绝大多数情况都是**检索召回范围太小**，关键内容没召回进来：

**解决方法/优化：**
- 多通道检索融合：同时用关键词检索（BM25） + 向量 embedding 检索两种方式，合并两个通道的召回结果，提高覆盖率
- 增加召回数量：原来只召回 top 3，改成召回 top 10，多召回几个候选，然后用重排序模型筛选出真正相关的
- 查询扩展（Query Expansion）：对用户原始查询做扩展，让 LLM 生成几个相关的同义查询，多个查询分别检索再合并结果，能召回更多相关内容

---

## 结果排序阶段

**本阶段核心：多路召回 → 选出最相关的 top-K**

### 20. 什么是 RAG-Fusion？它解决什么问题？核心技术和工作流程是什么？

**Q:** RAG-Fusion 原理是什么？

**A:**

**解决的问题：** 单一检索通道无法覆盖所有语义匹配方向，不同检索方式各有优缺点，融合起来能提高覆盖率。

**核心思想：** 融合多种不同检索通道的结果，发挥各种检索方式各自的优势，提高整体召回覆盖率和稳定性，最终让回答更全面更准确。

**核心技术：**
1. **多通道检索：** 同时用多种不同的检索方式（比如 embedding 向量检索 + keyword 关键词检索 + 知识图谱检索等），分别检索得到多组候选结果
2. **结果融合机制：** 对多个检索通道返回的候选结果做融合排序，常用方法是 RRF（Reciprocal Rank Fusion，倒数秩融合），然后把融合排序后的 top 结果交给 LLM 重新生成最终答案

**完整工作流程：**
1. **多个通道并行检索：** 不同检索通道（向量、关键词）分别检索，各自得到自己的候选排序结果
2. **汇总候选集合：** 把所有通道返回的候选合并在一起，去掉重复的
3. **加权融合排序：** 用 RRF（Reciprocal Rank Fusion，倒数秩融合）等方法对所有候选重新排序，RRF 公式：`score(d) = Σ 1 / (k + rank_i(d))`，融合多个排序结果得到更稳定的最终排序
4. **LLM 最终生成：** 把排序后的 top-k 结果上下文喂给 LLM，生成最终回答

---

### 21. RAG-Fusion 的优势与不足？

**Q:** RAG-Fusion 优缺点分析？

**A:**

**优势：**
- 召回结果更全面，能覆盖不同检索方式各自的优点
- 整体准确率更高，稳定性更好
- 有效减少漏检问题，一个通道漏了另一个通道可能找回来

**不足：**
- 成本更高：需要多次检索，消耗更多计算资源
- 延迟更大：并行检索也需要等最慢的通道出来，整体延迟比单通道大
- 工程实现复杂：需要维护多个检索通道，还要调优融合策略，比单通道复杂不少

---

## 上下文处理阶段

**本阶段核心：检索结果 → 构建 Prompt 上下文**

### 22. 什么叫"脱离上下文"问题？为什么多个文档拼接后容易出问题？具体怎么解决？怎么优化？

**Q:** 脱离上下文问题怎么理解？怎么解决？

**A:**

**什么问题：**
RAG 需要把检索到的多个相关文档片段，拼接起来塞进上下文窗口给 LLM。但如果只是简单粗暴把多个片段堆在一起，会出问题：
- 不同文档讲的是不同内容，甚至可能是相反的观点，拼接在一起后语义边界消失了
- LLM 分不清哪段内容是谁，哪个片段讲什么，容易混淆不同来源的信息
- 最后回答问题时，就会脱离检索到的上下文，模型还是用自己的知识瞎编

**举个实际例子：**
你问问题"新品A产品支持什么功能"，系统检索到了产品A文档和产品B文档，简单把两个文档拼在一起塞给 LLM，LLM 可能会把产品B的功能错当成产品A的回答给你，这就是典型的脱离上下文问题.

**解决方案/优化方法：**
- **Context Window Re-weighting（上下文窗口重加权）：** 对和问题更相关的片段，给更高的注意力权重，告诉 LLM "这段内容更重要，你要重点看"，让 LLM 聚焦在真正相关的内容
- **基于语义的有序拼接：** 不随便乱拼，按语义相关性从高到低排序，而且每个文档来源做清晰分隔（比如加上`[文档1]`、`[文档2]`标记），让 LLM 清楚看明白分段，不会混
- **上下文压缩：** 不是把整段都塞进去，只把文档中真正和问题相关的句子抽出来，扔掉不相关的内容，减少冗余和噪音，让 LLM 更容易抓重点

---

### 23. 明明已经检索到相关内容，但 LLM 就是没能给出正确答案，原因是什么？怎么解决？怎么优化？

**Q:** 检索对了但回答错了，什么原因？怎么解决？

**A:**

**常见原因：**
- 检索结果范围太广：召回了太多不相关内容，LLM 在这么多内容里找不到重点
- 检索结果范围太窄：召回太少，关键信息没覆盖到，缺胳膊少腿

**解决方法/优化：**
- 调整相似度阈值：阈值设太高召回太少，阈值太低噪音太多，需要根据你的数据集调优找平衡点
- 使用 Prompt 工程优化：明确给 LLM 指令要求，"你必须基于提供的参考内容回答，如果参考内容没有相关信息，请直接说明不知道，不要自己编造"，这样能减少 LLM 瞎编
- 做两步检索：第一步先粗召回，然后根据第一次检索结果 LLM 提取关键词，做二次检索补充，保证关键信息都覆盖到

---

### 24. 输出格式错误是什么原因？怎么解决？怎么优化？

**Q:** RAG 输出格式错误，不符合要求，怎么解决？

**A:**

**常见原因：**
- 源数据清洗不规范：原始文档进来的时候格式就混乱，带很多没用的格式噪音，索引进去之后污染了检索结果
- 没有对 LLM 输出做结构化约束：Prompt 没说清楚要什么格式，LLM 自然乱输出

**解决方法/优化：**
- 预处理阶段统一索引格式：所有文档进来都先清洗干净，统一格式之后再入库索引
- 在 Prompt 中明确约束输出模板：举个例子，明确要求"请用 JSON 格式输出答案"，或者"请分点回答，第一点...第二点..."，给 LLM 明确格式指令

---

### 25. RAG 在法律、医学这些特定专业领域效果特别差，容易出错，怎么优化？

**Q:** 特定领域效果差怎么优化？

**A:** 根本原因是：特定专业领域术语很多，通用预训练的 Embedding 模型对专业术语的语义表达不好：

**解决方向/优化方法：**
- 引入领域预训练词向量，或者用领域数据对 Embedding 模型做微调，让模型学会专业术语的正确表示
- 结合知识图谱增强：把领域概念、实体关系这些结构化知识注入检索过程，帮助模型更好理解专业问题
- 人工构建领域术语表，辅助分词和匹配，解决专业术语切分错误问题

### 26. 生成模块有哪些优化方法？

**Q:** 生成回答阶段怎么优化？

## 效果评估阶段

### 27. RAG 系统怎么评估效果？有哪些常用指标？

**Q:** 怎么评估 RAG 系统好不好？有哪些评估指标？

**A:** RAG 评估分为**检索阶段评估**和**生成阶段评估**两个层面：

**检索阶段评估（衡量检索对不对）：**
- **Recall@K（召回率）：** 正确的相关文档有没有被检索回来，正确文档在 top K 中的比例
- **Precision@K（精确率）：** 检索回来的 top K 文档中，真正相关的比例
- **MRR（Mean Reciprocal Rank）：** 衡量第一个正确文档出现的位置排名，排名越靠前分数越高
- **NDCG：** 考虑排序位置权重的衡量指标，更相关的文档排前面得分更高

**生成阶段评估（衡量回答好不好）：**
- **忠实度（Faithfulness）：** 回答是否只基于检索到的内容，有没有编造
- **相关性（Relevance）：** 回答是否真正对应用户问题，回答跑题了没有
- **正确率：** 人工或用大模型打分判断回答是否正确
- **BLEU/ROUGE：** 和标准答案对比的传统 NLP 指标，现在少用了

**实际工程中：** 通常离线先看 Recall@K 保证召回，在线做人工抽样评估回答的忠实度和正确性.

## 进阶方法篇

这些都是针对特定问题的进阶解决方案，面试高频考点：

### 28. 什么是 ParentDocumentRetriever？解决什么问题？原理是什么？

**Q:** ParentDocumentRetriever 是什么方法？

**A:**

**解决的问题：** 分块大小一直是矛盾：块太小检索准但信息不全，块太大信息全但检索不准.

**核心原理：** 父文档检索器用"两块切分"解决这个矛盾：
- 小 chunk（子文档）：用来做检索，小块语义更聚焦，检索更准确
- 大 chunk（父文档）：检索到小 chunk 后，把对应的完整大块父文档返回给上下文，保证上下文信息完整

**优势：** 兼顾了小块检索准确性和大块信息完整性，比单一固定大小分块效果好。LangChain 原生支持这个方法.

---

### 29. 什么是 MultiQueryRetriever？解决什么问题？原理是什么？

**Q:** MultiQueryRetriever 是什么？解决什么问题？

**A:**

**解决的问题：** 用户的问题可能表述模糊，或者有多个语义方向，单次检索召回不全.

**核心原理：** 让 LLM 基于用户原始问题，从多个角度生成几个不同的查询表述，然后每个查询分别检索，最后合并所有检索结果去重，召回更多相关内容.

**优势：** 能扩展查询的语义覆盖范围，提高召回率，解决用户提问表述模糊带来的漏检问题.

**劣势：** 需要多次调用 LLM 生成查询，成本和延迟都会增加.

---

### 30. 什么是 HyDE（Hypothetical Document Embeddings）？解决什么问题？原理是什么？

**Q:** HyDE 是什么原理？解决什么问题？

**A:**

**解决的问题：** 用户问题通常很短，而且和文档表述方式差异大，直接对用户问题做向量化检索效果不好.

**核心原理：**
1. 用户提问后，先让 LLM 凭空生成一篇"假设可能存在的回答文档"
2. 然后对这篇生成的假设文档做向量化，用这个向量去检索
3. 因为假设文档的长度和表述方式更接近真实库中文档，所以检索效果更好

**优势：** 解决了短查询和长文档分布不匹配的问题，对复杂问题检索效果提升明显.

**劣势：** 需要先生成假设文档，多了一步 LLM 调用，成本延迟增加.

---

### 31. 什么是 Self-RAG？和普通 RAG 有什么区别？

**Q:** Self-RAG 是什么？和普通 RAG 不同在哪里？

**A:**

**普通 RAG 的问题：** 不管什么问题都一定要检索，有时候其实模型自己已经会了，不需要检索，而且检索到不好的内容反而干扰.

**Self-RAG 核心思想：** 让 LLM 自己决定什么时候需要检索，检索到内容之后自己判断这个内容靠不靠谱，要不要用.

**工作流程：**
1. 收到问题后，模型先判断：这个问题我要不要检索？
2. 如果需要检索，就去检索，然后模型判断检索到的内容和问题相关吗？可用吗？
3. 最后基于判断，决定要不要用检索到的内容生成回答

**优势：** 自适应检索，不必要检索就不检索，减少了不必要的成本和干扰，比固定每次都检索效果更好.

---

### 32. 什么是 GraphRAG？和普通 RAG 有什么区别？

**Q:** GraphRAG 是什么？解决普通 RAG 什么问题？

**A:** GraphRAG 是微软研究院提出的进阶 RAG 方法.

**普通 RAG 问题：** 只能检索离散的文档片段，无法捕捉文档中实体之间的关系，回答复杂推理问题效果差.

**GraphRAG 核心思想：** 把知识库构建成知识图谱，节点是实体，边是实体关系，检索的时候不仅检索文本，还利用知识图谱的关系做推理，能找到隐式关联的信息.

**优势：** 对需要跨文档推理、发现实体关系这类复杂问题，效果比普通 RAG 好很多.

**劣势：** 构建知识图谱本身就有成本，工程复杂度更高.

---

### 33. 从系统架构层面来看，RAG 有哪些优化点？

**Q:** RAG 架构层面怎么优化？

**A:**

- **引入缓存层：** 对用户高频查询结果缓存起来（用 Redis / Milvus 缓存），相同问题再来直接返回缓存结果，减少重复计算，降低延迟
- **支持流式检索/流式生成：** 边检索边给 LLM，边输出结果给用户，提升用户体验，让用户能看到逐步输出
- **模块化部署：** 把索引构建模块、在线查询模块、LLM 生成模块这三个模块分开部署独立扩展，可以根据不同模块的负载分别扩容，节省成本
- **异步索引构建：** 更新索引的时候异步构建，不阻塞在线查询，用户查询不用等索引构建完

### 34. 各个模块分别有哪些优化策略？

**Q:** RAG 从检索到生成到存储，各个模块分别怎么优化？

**A:**

**检索模块优化：**
- **语义分段：** 不要按固定长度一刀切，按自然语义段落切分文档，保留完整语义，效果更好
- **多向量融合：** 对文档不同粒度（句子、段落、全文）都编码，多级检索融合，提高召回率

**生成模块优化：**
- **动态上下文选择：** 不是召回多少塞多少，只选出最相关的片段塞进上下文，扔掉没用的内容，减少噪音
- **Prompt 约束：** 明确要求模型必须基于上下文回答，给出清晰的输出格式模板

**存储模块优化：**
- **向量压缩：** 用 PCA 或者量化方法压缩向量维度，降低存储成本，提高检索速度
- **版本管理：** 支持知识库版本管理，可以回滚，方便更新出错的时候回滚

### 35. 备用模型降级机制是什么？为什么需要？

**Q:** 什么是备用模型机制？在 RAG 系统里有什么作用？

**A:**

- **机制设计：** 在系统层面设计降级策略，如果召回失败了（检索不到任何结果）或者 LLM 输出异常（比如输出被内容安全拦截了，或者输出格式错了多次重试都不对），就自动降级切换到备用小模型回答，或者返回预设的兜底回答
- **为什么需要：** 提高系统整体鲁棒性和可用性，避免因为主模型出问题导致整个服务不可用，用户完全得不到回答
- **适用场景：** 生产环境对外服务，对可用性要求高的关键业务系统

## 安全机制篇

### 36. RAG 面临哪些安全挑战？怎么解决？怎么防护？

**Q:** RAG 系统会遇到哪些安全问题？具体怎么防护？

**A:**

**常见安全挑战：**
- **Prompt 注入攻击：** 恶意用户通过构造特殊指令绕过系统限制，诱导模型泄露检索到的敏感信息，或者让模型执行不应该做的指令
- **越权访问：** 用户能检索到自己权限范围之外的文档，拿到不该拿到的信息
- **敏感数据泄露：** 检索结果中包含敏感信息，被 LLM 整合后输出给用户

**解决方案：**
- **输入过滤：** 在进入检索流程之前，检测和过滤恶意的 Prompt 注入指令，提前拦截
- **分级权限认证：** 文档都打标签做权限分级，用户也有权限等级，检索的时候只返回用户有权限看的文档，没权限的不检索
- **数据脱敏：** 敏感数据提前做脱敏处理，索引里面不存储原始敏感信息
- **输出检测：** LLM 生成回答之后，输出要做敏感信息检测，确认没有泄露敏感信息再返回给用户

## 发展展望篇

### 37. RAG 未来发展方向是什么？会和哪些技术融合？

**Q:** RAG 技术未来往哪个方向发展？有哪些热点研究方向？

**A:** RAG 未来主要发展方向是和更多技术融合，解决现有局限性：

1. **多模态 RAG：** 不只是处理文本，支持图片、表格、音频、视频多种模态数据的检索和生成，这是现在热门方向
2. **GraphRAG（知识图谱融合）：** 和知识图谱结合，利用知识图谱的结构化关系增强复杂问题推理能力
3. **Self-RAG：** 让模型自我评估检索结果对不对，自己决定这一步要不要检索，自我优化，自适应检索，减少不必要检索
4. **AgentRAG：** 和 Agent 架构结合，由 Agent 驱动多轮检索、工具调用，能解决需要多步推理的复杂问题
5. **强化 Agent 自主检索：** 让 Agent 自己决定什么时候需要检索，应该检索什么，分几次检索，不需要人工控制流程
6. **结合在线学习（Online Fine-tuning）：** 检索到新知识后实时增量更新模型，RAG + 在线微调结合，兼具两者优点

## 总结：核心考点速记

| 考点 | 核心结论 |
|------|---------|
| RAG vs SFT | RAG 是让模型查资料，SFT 是把知识记进模型脑子里 |
| RAG 最大优势 | 减少幻觉，答案可追溯，知识更新快，整体成本低 |
| RAG 最大问题 | garbage in garbage out，最终效果高度依赖检索质量 |
| 分块策略对比 | 固定长度 → 递归切分 → 语义切分 → 父-子分块，效果递增复杂度递增 |
| BM25 + 向量检索 | 精确关键词匹配 + 语义匹配互补，融合提升召回覆盖率 |
| 相似度选择 | OpenAI text-embedding-3 推荐用点积，向量归一化后 cosine = 点积 |
| 进阶方法速览 | ParentDocumentRetriever 解决分块矛盾，HyDE 解决短查询问题，MultiQuery 扩展查询，Self-RAG 自适应检索，GraphRAG 知识图谱增强推理 |
| 常见坑 | 分块策略不对，排序不好，多文档上下文拼接乱 |
| 进阶方案 | RAG-Fusion 多通道融合，Rerank 重排序 |
