整理自公开资料和 LLM/RAG/Agent 开发岗位面试常见问题，适合大模型应用开发岗位面试复习

## （我的思考）

> 这里用来记录你读完后的个人思考、总结、面试准备笔记

1. **Q**: RAG 检索时，向量归一化后，点积和 cosine 相似度结果一样吗？
   **A**: 完全一样。当向量做了 L2 归一化（模长=1），从公式推导来看点积结果等价于 cosine 相似度，但直接用点积计算性能更高，省去了除以模长的除法运算。

2. **Q**: RAG 场景为什么基本不使用 L2 距离？
   **A**: L2 距离对向量模长（文本长度）敏感，实际使用中不如 cosine 稳定，一般不推荐。

3. **Q**: 向量不归一化时，cosine 和点积区别是什么？
   **A**: 
   - cosine：只关心方向（语义相似度），排除句子长度影响，长短文本公平比较，**稳定性鲁棒性更好**，RAG 推荐用
   - 点积：同时受语义相似度和句子模长影响，**长文本天然得分偏高**，容易把更长但不相关的文本排前面，结果不稳定

4. **Q**: RAG 检索距离选型总结？
   **A**: RAG 一般优先选择 cosine 相似度；如果向量已经归一化，可以直接用点积等价替换，性能更好。

5. **Q**: 存储一亿级以上向量，有哪些检索优化策略？
   **A**: 
   - **选型层面**：不推荐 FAISS 这种单机检索库，全量放内存会爆，应该选支持**分布式分片**的向量数据库（比如 Milvus/Qdrant），支持分布式存储和元数据过滤
   - **索引选型**：选 **IVF** 而不是 HNSW：IVF 构建快、内存占用低，适合大规模数据；HNSW 构建慢、内存占用高，只适合千万级以内小数据量低延迟场景
   - **压缩优化**：用 SQ（scalar quantization）或者 PQ（product quantization）量化向量，大幅节省内存，虽然损失一点点精度，但能提高缓存命中率，最终检索速度更快
   - **算法层面**：肯定用 **ANN（近似最近邻）** 不用 KNN：亿级数据 KNN 是 O(N) 遍历，工程上完全不可行，速度极慢；ANN 用空间换时间，做到 O(log N) 复杂度，工程上可行，IVF/HNSW 都是 ANN 的主流实现

## 基础概念篇

### 1. 什么是向量数据库？为什么需要向量数据库？普通数据库不能存向量吗？

**Q:** 向量数据库和 MySQL 区别在哪？

**A:**

**向量数据库** 就是专门为**存储和检索向量嵌入**优化的数据库。

**为什么需要专门的向量数据库：**
- **核心需求是相似性检索**：给一个查询向量，找出 topk 最相似的向量
- 朴素 k-NN 遍历所有向量计算距离，O(n) 太慢，n 大了根本不能用
- 需要专门的**近似最近邻（ANN）索引算法**，加速检索，O(log n) 或者更快

**普通数据库不能干这个：**
- MySQL 只能做等于/范围查询，不能做高效相似性检索
- PostgreSQL 虽然有 pgvector 扩展能存向量，但数据量一大（百万级以上）检索速度就很慢，比专门向量数据库差很多

**总结：** 向量数据库 = 向量存储 + 高效ANN相似性检索 + 元数据过滤 + 增删改查，专门为 RAG/推荐等向量场景优化。

---

### 2. 向量距离常用哪些计算方法？什么时候用哪种？

**Q:** cosine 距离和 L2 距离区别是什么？

**A:**

常见距离计算：

| 距离 | 公式 | 适用场景 |
|------|------|----------|
| **Cosine 相似度** | `cosθ = (a·b)/(|a||b|)`，归一化后等价于 L2 | 对比文本嵌入，关注方向不关注模长，**RAG 最常用** |
| **Euclidean L2 距离** | 欧氏距离，空间几何直线距离 | 如果向量已经 L2 归一化，cosine 和 L2 等价 |
| **Dot Product（内积）** | `a·b` | 推荐系统，评分排序常用 |
| **Manhattan L1 距离** | 绝对值距离 | 高维向量鲁棒性好一点，不常用 |

**RAG 场景一般选 cosine 相似度**，因为文本嵌入主要关注语义方向。

---

### 3. 为什么向量检索都是近似最近邻（ANN）不能精确最近邻？

**Q:** 为什么要近似，不能准确找最近的吗？

**A:**

- 当向量数量是百万级甚至亿级，精确 k-NN 需要 O(N) 计算距离，太慢了，延迟扛不住
- ANN 用空间换时间，构建索引预处理，检索时候 O(log N) 或者 O(√N)，速度快很多
- 近似检索牺牲一点点准确率，换回来几十上百倍的速度提升，工程上完全可以接受
- 亿级向量要精确检索，延迟要几秒，用户体验根本受不了

所以工程上都用近似最近邻，准确率损失几个百分点，速度快很多，划算。

## 索引算法篇

### 4. 常见向量索引算法有哪些？各自优缺点是什么？

**Q:** 说一下 IVF, HNSW, SCaNN 区别？

**A:**

| 索引 | 全称 | 核心思想 | 优点 | 缺点 | 适用场景 |
|------|------|----------|------|------|------|----------|
| **IVF（Inverted File）** | 倒排文件 | 先聚类分桶，查询只搜最近几个桶 | 构建快，内存省，大数据量扩展性好 | 精度不如 HNSW，查询比 HNSW 慢 | 亿级向量，吞吐量优先 |
| **HNSW** | Hierarchical Navigable Small World | 分层导航图，跳表思想图检索 | 精度高，查询快 | 构建慢，内存占用大 | 百万到千万级，低延迟高准确 |
| **SCaNN** | Spill Sensitive Clustering-based Approximate Nearest Neighbor | 聚类+不对称距离计算 | 精度高，批量检索快 | 构建复杂 | 谷歌出品，大规模批量检索 |
| **FAISS** | 不是算法，是 FaceBook 开源的向量检索库，实现了 IVF/HNSW 等 | - | - | - | - |

**现在工业界常用就是 HNSW 和 IVF：**
- 数据量不大（千万级以内），要低延迟 → HNSW
- 数据量很大（亿级以上），吞吐量优先 → IVF

### 5. HNSW 原理是什么？为什么快？

**Q:** 讲讲 HNSW 分层图搜索。

**A:**

**HNSW = Hierarchical Navigable Small World**，基于图的索引，借鉴了跳表思想：

**分层结构：**
- 上层是稀疏连接的高速公路，方便长距离跳跃
- 下层是密集连接，方便精确定位
- 插入的时候按概率决定放哪一层

**搜索过程：**
1. 从最上层开始，找当前层最近邻
2. 走到下一层，继续找更近的
3. 重复直到最底层，输出最终结果

**为什么快：**
- 上层长距离跳跃减少了步数，不用每个节点都遍历
- 类似跳表，对数复杂度查找
- 查询速度比 IVF 快很多，精度也更高

**缺点：**
- 构建索引慢
- 内存占用大（每个节点存多个邻居指针）

---

### 6. IVF 原理是什么？IVF 里 nlist 和 nprobe 参数是什么意思？

**Q:** IVF 怎么工作？nlist nprobe 怎么调？

**A:**

**IVF = Inverted File** 倒排文件，聚类分桶思路：

**构建：**
1. 用 k-means 把所有向量聚类成 nlist 个簇
2. 每个向量分配到最近的簇中心
3. 倒排表记录每个簇里有哪些向量

**检索：**
1. 查询向量计算和每个簇中心距离，找出最近的 nprobe 个簇
2. 只在这 nprobe 个簇里面遍历找最近邻
3. 输出 topk

**参数调优：**
- `nlist`：聚类多少个桶 → 经验值 `nlist = 4 × sqrt(N)`，N 是总向量数
- `nprobe`：查询搜几个桶 → nprobe 越大，精度越高越慢
- 例子：百万级向量，nlist=1024，nprobe=8~64

**trade-off：** nprobe 大 → 精度高，速度慢；nprobe 小 → 速度快，精度低。

### 7. 量化对向量检索有什么用？PQ 是什么？

**Q:** 向量索引也需要量化吗？PQ 原理是什么？

**A:**

**为什么要量化向量：**
- 原始向量每个 float 4 字节，1 亿 128 维向量就是 1 亿 × 128 × 4 字节 = 512MB，其实还能压
- 量化减少存储，内存占用少，缓存命中率高，速度更快

**PQ（Product Quantization）乘积量化：**
- 把一个高维向量切成几段，每一段聚类量化，每个段用聚类中心代替
- 这样原来 128 维 float 只要几个 byte 就能存，压缩 4~8 倍
- 检索时候可以用量化后的中心计算距离，不用解码

**现在常用：**
- **Scalar Quantization（SQ）**：每个分量缩放到 8bit，简单高效，现在很多地方用这个，压缩 4 倍，精度损失小
- PQ 压缩比更大，精度损失也比 SQ 大一点

## 产品对比篇

### 8. FAISS 是什么？它是完整向量数据库吗？

**Q:** FAISS 和 Milvus 区别是什么？

**A:**

**FAISS** 是 Facebook 开源的**向量检索算法库**，实现了各种 ANN 索引（IVF/HNSW/PQ），只管检索，不管存储、增删改、元数据过滤。

**对比：**

| FAISS | Milvus/Qdrant |
|-------|--------------|
| 只是算法库，需要自己封装 | 完整数据库，提供 API，管存储、增删改、过滤 |
| 自己做持久化 | 自带持久化 |
| 自己做分布式 | 原生支持分布式 |
| 适合嵌入式，或者简单场景 | 适合生产环境复杂场景 |

所以：
- 你做原型验证，可以 FAISS + 自己存
- 生产环境用完整向量数据库（Milvus/Qdrant）

### 9. Milvus vs FAISS vs Qdrant vs Chroma 对比，怎么选？

**Q** 几个主流向量数据库各有什么特点？

**A:**

| 产品 | 公司/社区 | 特点 | 适合场景 |
|------|-----------|------|----------|
| **FAISS** | Meta | 纯算法库，高性能，啥都要自己搭 | 原型、嵌入式、简单场景 |
| **Milvus** | Zilliz 公司 | 功能完整，分布式，支持元数据过滤，生态成熟，功能丰富 | 企业生产环境，大数据量，分布式 |
| **Qdrant** | Qdrant 公司 | Rust 写，性能不错，原生支持矢量量化，云原生友好 | 云原生生产环境，对 RAG 友好 |
| **Chroma** | Chroma 社区 | Python 写，轻量，开箱即用，集成 LangChain 好 | 开发测试原型，小数据量，不适合大规模生产 |
| **Weaviate** | Weaviate 公司 | 原生多模态，GraphQL API | 多模态场景 |
| **Pinecone** | Pinecone 公司 | 全托管云服务，不用自己运维 | 云上使用，不想自己运维 |

**选型经验：**
- 开发测试原型 → Chroma
- 自己运维生产 → Milvus / Qdrant
- 云托管不想运维 → Pinecone
- 大数据量分布式 → Milvus

### 10. Milvus 索引怎么选？什么时候用 HNSW，什么时候用 IVF？

**Q:** Milvus 中 INDEX_TYPE 怎么选？

**A:**

经验规则（Milvus 官方推荐）：
- **数据量 < 1M 向量** → **HNSW**，精度高，查询快
- **数据量 > 1M 向量** → **IVF**，构建快，省内存，吞吐高
- **数据量 > 10M 向量** → **IVF** 或者 SCaNN

trade-off 再提醒：
- HNSW：精度高查询快 → 构建慢吃内存
- IVF：构建快省内存 → 查询精度速度不如 HNSW

### 11. Qdrant 优势是什么？和 Milvus 区别？

**Q:** Qdrant 好在哪？

**A:**

**Qdrant 主要特点：**
1. Rust 实现，性能不错，内存安全
2. 原生支持量化压缩，压缩比高，精度损失小
3. 云原生友好，Kubernetes 友好
4. 内置语义负载均衡，分布式支持
5. 对 RAG 场景优化比较好

**区别：**
- Milvus 发展早，生态更成熟，功能更多，分布式成熟
- Qdrant 后起之秀，Rust 架构新，云原生友好，用起来更简洁
- 两个现在都很不错，看你技术栈偏好

## 工程实践篇

### 12. RAG 中向量检索召回后为什么要重排序（Rerank）？

**Q:** 召回已经有结果了，为什么还要重排序？

**A:**

**两个原因：**
1. **向量检索是暴力语义匹配**：embedding 是全局语义，有时候第一不是最相关的，重排序用 Cross-Encoder 更精准地计算 query-doc 相关性，把真正相关的排到前面
2. **提高召回率，降冗余**：向量召回一般回 10~50 个，重排序选 top3~top5 给大模型，既提高了前几名准确率，又不占太多 context 窗口

**收益：** 重排序能提升 RAG 准确率 10%~30%，成本就是多一次模型调用，性价比很高，现在工业界 RAG 基本都加这一步。

---

### 13. 向量数据库怎么更新？增删改支持好吗？

**Q:** HNSW 支持动态增删吗？

**A:**

- **IVF**：支持增删，添加方便，删除要做墓碑
- **HNSW**：现代实现（Milvus/Qdrant/FAISS）都支持动态增删了，添加很快，删除是逻辑删除
- 都支持增量插入，不用重建整个索引，生产够用

---

### 14. 向量数据库支持过滤吗？怎么过滤？

**Q:** 我要先按元数据过滤（比如时间范围），再搜向量，向量数据库支持吗？

**A:**

都支持**前置过滤**：先按元数据过滤出分段，再在分段里面做向量检索。

实现方式：
- Milvus 用 `expr` 写过滤条件，支持 `=/>/</in/between`
- Qdrant 用 must/should 过滤，支持布尔条件
- 过滤会减少检索向量数量，速度会快一点，精度不影响

---

### 15. 海量向量（亿级以上）怎么部署？

**Q:** 一亿以上向量怎么存怎么搜？

**A:**
1. **用 IVF 索引**，比 HNSW 省内存
- 2. **量化**：用 SQ 8bit 或者 PQ 压缩，省内存
- 3. **分布式**：Milvus 支持多分片，把数据分到多个节点
- 4. **分层检索**：先粗召回再精排

亿级向量现在成熟方案就是 Milvus 分布式 + IVF + 量化，能扛住。

---

### 16. RAG 里对长文档怎么处理向量存储？

**Q:** 长文档分块后每个块一个向量吗？

**A:**

主流做法：
1. **分块**：把长文档切分成 512~1024 token 的块
2. **每个块单独 embedding** 得到一个向量
3. **存在向量数据库**，元数据带上文档 ID、块位置
4. 检索时候找到相关块，把块内容给大模型回答

**进阶方法：**
- **父子分块**：父块大一点，子块小一点，存子块向量，召回后拿父块给 LLM，兼顾召回精度和上下文完整性
- **Multi-vector**：一个文档多个向量，不同粒度

## 总结速记

| 考点 | 核心结论 |
|------|---------|
| 为什么需要向量数据库 | 专门优化近似最近邻相似性检索，MySQL/pgvector 大数据量太慢 |
| 距离计算 RAG 用什么 | Cosine 相似度，归一化后等价 L2 |
| HNSW | 分层导航图，精度高查询快，构建慢吃内存，适合千万级以内低延迟 |
| IVF | 聚类分桶，nlist 是桶数，nprobe 是查询搜几个桶，适合亿级以上吞吐优先 |
| 量化 | 压缩存储，减少内存，提高缓存命中率，加快检索，SQ 8bit 简单好用 |
| FAISS vs Milvus | FAISS 是算法库，Milvus 是完整向量数据库 |
| 选型：开发测试 → Chroma，自研生产 → Milvus/Qdrant，云托管 → Pinecone |
| RAG 为什么要重排序 | 提高前几名精度，给 LLM 最好的结果，省 context 空间 |
| HNSW vs IVF 选择 | 千万以内低延迟 → HNSW，亿级吞吐 → IVF |
