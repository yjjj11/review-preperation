# AutoModerate 开发思考 FAQ

记录项目开发过程中的技术选型思考与决策 rationale。

---

## Q1: 为什么一审选用 Qwen2.5-0.5B，而不是传统分类器/CNN/LSTM/更小的模型？

### 角度一：泛化能力对比

**为什么不选传统机器学习（XGBoost + TFIDF/关键词）？**
- 内容审核场景下，违规表达方式一直在变异（比如"薇❤"、"V❤"、"卫星"都指代微信）
- 传统方法依赖人工提特征，泛化能力差，变异词识别不了，规则维护成本高
- 大模型通过预训练学到了丰富的语义知识，能理解谐音、替代写法，泛化能力强很多

**为什么不选 CNN/LSTM 小模型？**

**本质区别：理解句子的方式不同**
| 模型 | 理解方式 | 通俗比喻 |
|------|----------|----------|
| **CNN** | 只看局部窗口里的词（一次看2-3个），层层卷积 | 用放大镜一行一行扫文章，只能看到眼前几个字，看不到整句关系 |
| **LSTM** | 从左到右串行读，把前面信息带到后面 | 顺序读文章，读后面时前面的信息已经忘得差不多 |
| **Transformer (Qwen)** | 每个词直接看所有其他词，注意力算相关性 | 一眼看完整个句子，瞬间知道哪个词和哪个词有关系 |

- CNN/LSTM 的长程依赖建模能力弱，对整句语义理解不如Transformer
- 最大差距来自**预训练**：CNN/LSTM 需要从零开始训练，所有知识都只能从你给的训练数据里学；Qwen2.5 在万亿级中文互联网预训练过，已经见过各种表达方式，泛化能力强很多——"薇❤"这种谐音变体，CNN/LSTM 没见过就学不会，大模型能理解
- 每个检测任务（辱骂/广告/刷单/事实核查）需要单独训练一个模型，工程上更繁琐
- **没法用 prompt 做多任务共享**，不能一个基座搞定四个任务

**什么叫做"用 prompt 做多任务"？**
```
# 同一个 0.5B 基座，只需要改 prompt 就切换不同检测任务：
prompt_1 = "你是专业的辱骂内容检测器，判断以下评论是否包含辱骂：{comment}"
prompt_2 = "你是专业的广告引流检测器，判断以下评论是否包含广告：{comment}"
# ... 四个任务共用一个模型权重，只需要输出置信度
```
- 四个任务共享同一个基座，显存只需要放一份模型
- 新增检测任务只需要写新 prompt + 微调对应 LoRA，不用重新训整个基座

### 角度二：尺寸选择权衡（为什么是 0.5B，不是更小）

| 尺寸 | 优点 | 缺点 |
|------|------|------|
| **0.5B** | 精度足够，vLLM 推理单条仅 0.03s，满足高并发 | - |
| <0.3B | 模型更小更快 | 语义理解能力掉太多，置信度输出不可靠，一审准确率下降会导致二审触发率上升，反而更费算力 |

结论：0.5B 是"速度够用，精度够用"的甜蜜点。

### 角度三：为什么是 Qwen2.5，不是其他开源小模型？

1. **中文原生优化**：阿里开源，中文理解比 Llama 同尺寸好很多，对中文评论审核更有利
2. **同尺寸效果领先**：Qwen2.5 在 0.5B/1.5B 区间效果优于其他开源模型
3. **许可友好**：Apache 2.0，商用无限制
4. **生态完善**：vLLM 支持好，各种量化格式官方都提供，开箱即用

## Q2: 既然 0.5B 已经预训练过了，为什么还要微调？微调有什么用？

**预训练和微调目标完全不同，两者缺一不可**：

| 阶段 | 核心目标 |
|------|----------|
| **预训练** | 让模型学会中文，掌握语法、语义、常识，具备泛化能力，能识别各种谐音变异违规词 |
| **微调** | 将通用知识对齐到你的具体任务，让模型学会输出违规置信度 |

**微调带来两点核心提升**：
1. **输出格式对齐**：预训练是通用文本生成，微调让它输出 [0,1] 置信度，适配二分类任务
2. **准确率提升**：预训练对边界案例（如"商家微信客服"算不算广告）判断模糊，微调教会模型你的判定标准，决策边界更清晰

**LoRA 优势**：
只训一小部分参数，基座保留预训练泛化能力；每个任务一个独立 LoRA 权重，**可插拔**——新增任务只需训新 LoRA，基座不动；多个任务共享一份基座显存，节省资源。

**一句话结论**：预训练给泛化，微调给对齐，LoRA 给可插拔多任务能力。

## Q3: 为什么把二审模型从云端改到本地部署？

主要有三个原因：

1. **速度更快**：本地部署省去了公网API调用延迟，二审推理延迟从数百ms降到几十ms，整体响应更快
2. **成本更低**：云端大模型API按调用次数付费，业务量起来后成本很高；本地一次性部署，长期使用成本低很多
3. **数据隐私与稳定性**：电商评论属于用户生成内容，本地部署数据不出域，更合规；不依赖第三方云端服务，可用性自己可控，不会被限流、服务降级影响稳定性

我们通过 AWQ 4bit 量化，让 7B 模型能在 8GB 显存运行，技术上具备了本地部署可行性，因此选择本地部署。

## Q4: 为什么要设计两级检测架构，不用一个大模型直接检测？

核心是**速度、成本、精度的权衡**：

| 方案 | 速度 | 成本 | 精度 |
|------|------|------|------|
| **一级大模型直接检测** | 慢，每条都要7B推理 | 高，显存占用大，吞吐量低 | 高 |
| **两级架构（小模型一审 + 大模型二审）** | 快，90%+ 请求0.03s返回 | 低，大模型只处理低置信度请求，吞吐量是纯大模型的数倍 | 高，低置信度才会到大模型 |

设计思路：
- 大部分评论都是明显合规或明显违规，小模型就能搞定
- 只有少数边缘低置信度案例需要大模型深度分析
- 这样既保证了整体低延迟高吞吐量，又保证了最终准确率，还省下了推理成本

## Q5: 为什么四个检测任务并行执行，而不是串行或者合并成一个任务？

### 为什么不串行？
四个检测任务（辱骂/广告/刷单/事实核查）是**相互独立**的，不存在依赖关系：

- 串行：总延迟 = 四个任务延迟相加 → 慢
- 并行：总延迟 = 四个任务中最慢的那个的延迟 → 快，不随任务数累加

LangGraph 原生支持并行节点，我们利用这个特性做任务并行，把总 latency 压到最低。

### 为什么不合并成一个任务，一次性输出四个结果？就算用 prefill 缓存公共前缀也不行吗？

即使利用 prefill 缓存公共前缀，问题依然存在：

1. **计算量还是更大**：
   - 虽然公共前缀可以缓存，但整个输入序列长度还是比单个任务长很多，prefill 计算量和内存访问量还是更大
   - 输出需要同时生成四个置信度，生成长度也从 1~2 个 token 变成 4~8 个，解码步数增加

2. **准确率会下降**：
   - 模型需要同时在一个前向里输出四个任务结果，任务之间会互相干扰
   - 我们的实际方案是：**同一个基座 + 同一个LoRA，通过不同prompt区分四个任务**，训练数据混合了四种任务样本，模型通过prompt切换任务上下文；拆分推理保证每个任务的上下文干净，准确率更高

3. **显存与工程更优**：
   - 一个LoRA搞定四个任务，不需要同时加载多个，已经是最省显存的拆分方式
   - 新增/修改某个检测任务，只需要补充对应训练数据，增量微调LoRA即可，基座不动，迭代灵活

顺便澄清一个问题：一个模型能同时插多个LoRA并运行吗？
- 现在推理框架（如vLLM）支持**多LoRA动态加载**，就是服务里同时存多个LoRA，不同请求用不同LoRA
- 但**同一个推理请求里，只能激活一个LoRA**，并不能一次前向同时用多个LoRA输出四个结果

所以我们"**同一个基座+同一个LoRA+不同prompt区分任务**"的方案，就是兼顾了显存省、准确率、工程简单，是很合理的设计。

所以结论是：就算有 prefill 缓存优化，**拆分prompt+并行推理**在速度、精度、工程迭代上都更优。

## Q6: 为什么 Milvus 选择用 Milvus Lite，不用 standalone 容器部署？

**对中小规模场景来说，Milvus Lite 体验更好**：
- 零依赖：不需要 Docker，不需要启动独立服务，就是一个本地文件
- 部署简单：pip install 就能用，项目拉下来直接跑
- 数据量匹配：你的项目RAG文档数量不会特别大，Milvus Lite 性能完全够用
- 如果后续业务增长，切换到 standalone 分布式也很方便，接口兼容

所以选择 Milvus Lite 是**工程易用性**和**功能需求**之间的平衡，降低部署门槛，适合 demo 和中小规模生产。

## Q7: 为什么一审模型只做 FP8 KV缓存量化，不做 AWQ 权重量化？

### 核心区别
| 量化方式 | 量化对象 | 解决什么问题 |
|----------|----------|--------------|
| **AWQ/GPTQ** | 模型权重 | 减少权重体积，节省权重占用显存 |
| **FP8 KV缓存量化** | 推理生成的KV缓存 | 减少每个请求预留的KV空间，节省运行时显存 |

### 为什么不做 AWQ 权重量化？

1. **收益低**：0.5B FP16 权重才 ~1GB，AWQ 4bit 后 ~300MB，省几百MB对整体收益不大，0.5B 本来就能放下
2. **实际反而变慢**：
   - AWQ 需要推理时**反量化**：GPU 不支持 4bit 原生计算，必须先解压回 FP16 才能计算，有额外开销
   - **尺寸效应**：大模型权重体积大，读显存是瓶颈 → 压缩后读省的时间 > 反量化开销 → 变快；小模型权重小，读显存不是瓶颈 → 反量化开销占比更高 → **反而变慢**（实测验证）
3. **精度优先**：一审是第一道关卡，尽量保留权重原始精度更安全

### 为什么 FP8 KV量化非常适配？

1. **显存瓶颈本来就是KV缓存**：高并发下，KV缓存总占用比权重更大，优化KV缓存收益远大于权重量化
2. **无额外开销**：FP8 是 NVIDIA Ada+ GPU 原生支持，不需要反量化，直接计算，没有额外开销
3. **精度可接受**：一审只输出 1~2 个 token，对 KV缓存精度要求不高，FP8 压缩损失可忽略
4. **提升并发**：FP8 能省出一半显存，显存更多就能同时放更多请求的KV缓存 → **支持更高并发，提升整体吞吐量，避免OOM**

**结论**：一审权重本身小，KV缓存才是显存瓶颈；小模型AWQ反量化开销反而拖慢速度；FP8 KV量化原生支持无额外开销，精度损失可忽略，能省出显存支持更高并发。因此选择该方案，性价比最高。

## Q8: AWQ/GPTQ（权重量化） vs FP8（KV缓存量化）对比总结

| 维度 | AWQ/GPTQ | FP8 KV缓存量化 |
|------|----------|----------------|
| **量化对象** | 模型权重（只读，一次加载永久存在显存） | 推理时动态生成的KV缓存（每个请求都有） |
| **需要反量化吗？** | 需要（GPU不支持4bit原生计算） | 不需要（NVIDIA Ada+ 原生支持FP8计算） |
| **有无额外开销？** | 有，反量化需要额外计算 | 无，直接计算 |
| **解决什么问题** | 减少权重占用的显存，让大模型能塞进显卡 | 减少每个请求预留的KV空间，支持更高并发 |
| **适合场景** | 大模型（7B+）权重压缩，让大模型能在有限显存跑 | 高并发推理，省出KV空间放更多请求，提升吞吐量 |
| **对小模型0.5B** | 收益低，反量化开销占比高，反而变慢 | 收益高，KV缓存是瓶颈，省出显存给并发 |

两者**完全可以同时用**，不冲突：
- AWQ 作用在**模型权重**，FP8 作用在**动态生成的KV缓存**，是不同的数据
- GPU不支持4bit原生计算只影响权重，需要对权重做反量化；KV缓存用FP8是GPU原生支持，直接计算
- 你的二审 7B 模型就是这个组合：`AWQ 4bit 权重 + FP8 KV缓存`，权重体积小，KV缓存也省空间，是显存有限情况下的最优搭配

## Q9: 为什么不把 vLLM 换成 TensorRT-LLM 追求更快速度？

TensorRT-LLM 确实在特定场景下更快，但不适合我们的情况：

- **收益不对等**：TensorRT-LLM 对 **7B+ 大模型、长序列输入** 的优化效率更高；我们一审是 0.5B 小模型，已经做到 0.03s/条，就算再快 20% 也只是 0.024s，用户完全感知不到，边际收益极低
- **工程成本高**：需要离线编译模型，改模型结构或换模型就要重编译，不如 vLLM 开箱即用支持动态长度
- **本地硬件限制**：我们是本地 8GB 显存部署，vLLM 的 PagedAttention + 动态批处理已经满足高并发需求，换 TensorRT-LLM 得不到明显好处

**结论**：TensorRT-LLM 在大模型长序列场景确实更快，但我们小模型+本地8GB显存部署场景下，vLLM 已经够用，换了边际收益极低，徒增工程成本，因此不换。

## Q10: 为什么用 MSE + LoRA 微调，不用 DPO？

DPO 确实适合偏好优化，我们**天然就有 DPO 需要的数据**：**一审预测 vs 二审修正**就是天然的偏好对（二审结果 > 一审结果）。但当前方案已经足够好：

| 维度 | 现有方案（MSE + LoRA） | DPO |
|------|------------------------|-----|
| 训练目标 | 拟合二审给出的置信度 | 对齐二审的偏好 |
| 实现复杂度 | 简单直接，已经跑通闭环 | 需要额外构造偏好对，实现更复杂 |
| 已验证效果 | 二审触发率下降 **40%**，收益明确 | 理论上可能进一步降低，但边际收益递减 |

**结论**：现有方案简单有效，已经拿到明确收益，DPO 是锦上添花的进阶优化，不是必须。当前方案在"效果/工程成本"之间已经取得很好的平衡，因此保持现状。
